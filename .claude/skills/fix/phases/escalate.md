# Fix Phase — 升级（Escalate）

> **来源**：`when-stuck` skill 核心内容提取
>
> **用途**：`fix` skill 升级条件的执行参考。被 `fix` SKILL.md 升级条件触发时调用。
>
> **触发条件**：`fix` 的 `attempt_count ≥ 2` 且验证仍然失败。

---

## 触发条件（必须同时满足）

- 同一 bug / 同一问题，已尝试修复 **2 次**但未成功
- 用户回复"还是不行" / "又报了新错" / "不对" **2 次**及以上
- 或 AI 自己判断"我陷入循环了"

---

## fix-context 读取指令

**优先从** `${config.fix.context_file}`（默认为 `.claude/tmp/fix-context.yaml`）**读取前两次记录**。

**仅当**文件不存在时，才要求人工输入前两次的：
- hypothesis（假设根因）
- fix_plan（修复方案）
- result（失败原因）
- failed_evidence（什么证据表明假设错了）

**fix-context 格式说明**：
```yaml
session_id: "{uuid}"
attempt_count: 2
attempts:
  - attempt_id: 1
    hypothesis: "{假设根因}"
    fix_plan: "{修复方案}"
    result: "{失败描述}"
    failed_evidence: "{什么证据表明假设错了}"
  - attempt_id: 2
    hypothesis: "{新假设根因}"
    fix_plan: "{新修复方案}"
    result: "{失败描述}"
    failed_evidence: "{什么证据表明假设错了}"
```

---

## 强制分流流程

### 步骤 1：记录前两次假设

必须显式输出前两次的根因假设和失败原因：

```
## 两次失败记录

### 第 1 次尝试
**假设根因**: {假设是什么}
**修复方案**: {做了什么修改}
**结果**: {失败原因}
**证据**: {什么证据表明假设错了}

### 第 2 次尝试
**假设根因**: {假设是什么}
**修复方案**: {做了什么修改}
**结果**: {失败原因}
**证据**: {什么证据表明假设错了}
```

### 步骤 2：分析为什么失败

检查以下可能性：
- **假设本身错了** — 根因找错了位置
- **证据解读错了** — 看起来像 A，实际是 B
- **修复方案不完整** — 部分修复了，但没覆盖全部问题
- **环境/上下文遗漏** — 少了某个重要信息（如版本、配置、数据状态）
- **测试本身有问题** — 复现步骤或断言不对

### 步骤 3：选择新方向

| 新方向 | 适用场景 |
|--------|---------|
| **调用 debugging skill 重新分析** | 假设错了，需要重新走根因调查 |
| **调用 verification-before-completion** | 需要先确认修复到底有没有生效 |
| **多角度并行 Agent** | 问题复杂，需要同时排查多个方向 |
| **请用户提供更多信息** | 缺少关键上下文（日志/截图/复现步骤） |
| **回滚变更** | 修复引入了新问题，应退回上一版本 |
| **转 RPIV** | 问题范围超出 `/fix` 能力，需走完整流程 |

### 步骤 4：输出分流决策

```
## 🚦 重新分流决策

### 失败原因分析
{一句话说清前两次为什么失败}

### 新方向
**选择**: {从上方选择一个}
**理由**: {为什么这个方向更可能成功}
**具体行动**: {新方向的第一步是什么}

请确认是否同意此方向？
```

### 步骤 5：执行新方向

用户确认后，按新方向执行。

## RPIV 交接包

当 when-stuck 建议转 RPIV 时，将 `fix-context.yaml` 内容转换为交接包写入约定位置：

**路径**：`${config.fix.escalation_log}`（默认为 `.claude/tmp/fix-escalation.yaml`）

**内容**：
```yaml
escalation_type: rpiv_handoff
escalated_at: "{iso_timestamp}"
original_problem: "{用户描述的问题}"
failed_attempts: 2
last_diagnosis:
  root_cause: "{最后一次根因}"
  fix_plan: "{最后一次方案}"
  impact: "{影响范围}"
summary: "{一句话总结问题本质 + 建议走 RPIV 的理由}"
```

---

## 硬约束

- **禁止**第 3 次按同思路小参数调整就说是"新尝试"
- **禁止**没说明"前两次假设是什么、为什么错"就继续改
- **禁止**连续两轮用相同维度的问题
- **必须**每次重新分流都要等用户确认方向
- **必须**输出两次失败的完整记录后再提新方向

---

## 反合理化表格

| 借口 | 现实检查 |
|------|----------|
| "这次改法不一样" | 根因假设一样吗？一样 → 禁止 |
| "应该能工作了" | 有验证证据吗？没有 → 禁止 |
| "用户测过了" | 测的是修复后的版本吗？不确定 → 禁止 |
| "只是边缘情况" | 能证明不是根因吗？不能 → 禁止 |

---

## 输出格式

```markdown
## 🔄 重新分流

### 两次失败记录
| 次数 | 假设 | 修复 | 结果 |
|------|------|------|------|
| 第1次 | {假设} | {修改} | {失败} |
| 第2次 | {假设} | {修改} | {失败} |

### 失败原因
{分析为什么两次都失败}

### 新方向
{选择的方向 + 理由 + 第一步行动}

⚠️ 等待确认后按新方向执行。
```
