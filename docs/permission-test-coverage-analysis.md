# Permission模块分层测试覆盖分析报告

> **文档编号**: DOC-TEST-001  
> **更新时间**: 2026-03-12  
> **分析对象**: Permission 模块测试资产与当前测试门禁  
> **分析范围**: 前端 E2E、前端单测、后端单测/集成测试、E2E 门禁脚本  
> **分析目标**: 判断当前“一人公司开发框架”在测试覆盖上的真实问题，并给出可执行整改方向

---

## 执行摘要

### 核心结论
- 当前问题不是“Permission 模块没有测试”，而是**测试分层失衡**。
- **认证 / 找回密码**链路的覆盖深度相对较好，已经同时具备前端 E2E、前端单测、后端单测，以及部分真实行为证据。
- **角色 / 用户 / 基础数据 / 仪表盘**四个后台页面的 E2E 仍明显偏浅，很多用例停留在“页面能打开 + API 成功 + 表格可见”。
- 当前框架的门禁已经能强制“跑 E2E 命令”，但**不能证明测试场景完整**，也不能证明 `REQ-PER-001` 这种权限矩阵已经被真正覆盖。

### 对框架问题的准确表述
- 准确说法：**框架已解决“有没有测试命令”的问题，但尚未解决“测试是否覆盖关键语义”的问题。**
- 不准确说法：**Permission 模块整体测试覆盖几乎为空。**

### 风险判断
- 🔴 高风险: 权限矩阵、数据边界、CRUD 结果校验、业务规则拒绝场景
- 🟡 中风险: E2E 门禁只校验最低质量，不校验覆盖完整性
- 🟢 低风险: UI 契约和视觉一致性。这一层已经有较完整的合同与回归测试

### 本报告的口径约束
- 本报告**不再引用缺少运行报告附件的历史总数**，例如“49 通过，1 跳过”这类不稳定指标。
- 本报告只陈述当前仓库中可回读的测试资产与门禁能力。

---

## 证据基础

本次分析按 4 层回读现有仓库：

| 层级 | 主要证据 | 结论方向 |
|------|----------|----------|
| 前端 E2E | `osg-frontend/tests/e2e/*.spec.ts` | 判断用户路径、接口交互、页面后置结果是否被覆盖 |
| 前端单测 | `osg-frontend/packages/admin/src/__tests__/*.spec.ts` | 判断表单规则、交互逻辑、界面结构约束是否被覆盖 |
| 后端单测/集成测试 | `ruoyi-admin/src/test/java/...` | 判断接口契约、匿名访问、安全链路、控制器返回结构是否被覆盖 |
| 门禁与框架 | `bin/e2e-api-gate.sh`、`.claude/skills/workflow-engine/tests/e2e_api_guard.py` | 判断框架是否能把“测试存在”升级为“测试充分” |

## 第一层：前端 E2E 覆盖分析

### 1. 已经做得比较深的区域

#### 1.1 登录认证
当前登录 E2E 已经不只是“能登录”：
- 登录成功后校验跳转到 `/dashboard`
- 校验 token 已写入 `localStorage`
- 校验匿名登出被拒绝
- 记录行为契约结果，区分 `expected_result` 与 `observed_result`
- 校验超级管理员菜单可见性

这说明认证链路至少具备了：
- 页面行为验证
- 接口业务码验证
- 会话状态验证
- 菜单可见性验证

#### 1.2 忘记密码
忘记密码链路是当前 Permission 模块中覆盖最深的一块：
- 4 步完整流程跑通
- 通过 provider log 读取真实验证码证据
- 校验 reset token 返回
- 校验 UI 步骤切换到正确状态
- 校验无效验证码 / 过期验证码 / 无效 token / 重复错误消息去重
- 记录行为契约报告

这一块已经明显超出了“表面冒烟”的范畴。

### 2. 仍然偏浅的区域

#### 2.1 角色管理
当前 E2E 主要验证：
- 页面可访问
- `/api/system/role/list` 返回成功
- `/api/system/menu/treeselect` 返回成功
- 表格可见

缺失的核心语义包括：
- 新增角色后列表中是否出现新角色
- 分配权限后菜单/接口是否立即生效
- 有员工绑定的角色删除是否被真实业务规则拒绝
- 非超管角色是否被正确限制

#### 2.2 用户管理
当前 E2E 主要验证：
- 页面可访问
- 用户列表接口成功
- 角色选项接口成功
- 表格可见

缺失的核心语义包括：
- 创建用户后数据是否持久化
- 用户启用/禁用后登录能力是否变化
- 角色变更后菜单/权限是否即时变化
- 编辑态用户名只读是否真的通过表单交互被验证

#### 2.3 基础数据
当前 E2E 主要验证：
- 页面可访问
- 列表接口成功
- 搜索框存在

缺失的核心语义包括：
- 新增 / 编辑 / 禁用 / 启用是否真正改动数据
- 关联字段依赖是否被正确拦截
- 删除或状态变更是否影响上下游

#### 2.4 仪表盘
当前 E2E 已验证多个接口能返回成功，也验证了卡片和区块可见，但仍缺：
- 统计值是否与后端数据语义一致
- 不同角色看到的数据是否被过滤
- 点击卡片后的去向是否正确
- 异常/空态/大数据量是否正确处理

### 3. 对 E2E 层的判断

#### 3.1 真实结论
- E2E 层**不是空白**
- E2E 层**深度极不均衡**
- 当前 E2E 的主要薄弱点集中在：
  - CRUD 后置结果
  - 权限边界负例
  - 业务规则拒绝路径
  - 数据持久化与即时生效

#### 3.2 不能得出的结论
- 不能说“Permission 模块没有 E2E”
- 不能说“所有 E2E 都只测 HTTP 200”

更准确的说法是：**认证类 E2E 已经偏深，后台 CRUD 页面 E2E 明显偏浅。**

## 第二层：前端单测覆盖分析

### 1. 当前已有的单测资产

Permission 相关前端单测并不少，至少包括：
- `login.spec.ts`
- `forgot-password.spec.ts`
- `permission-guard.spec.ts`
- `permission-menu.spec.ts`
- `roles.spec.ts`
- `users.spec.ts`
- `base-data.spec.ts`
- `dashboard.spec.ts`

### 2. 当前单测覆盖的内容

#### 2.1 忘记密码
前端单测已覆盖：
- 邮箱格式边界
- 验证码长度边界
- 密码复杂度边界
- 步骤切换逻辑
- 错误消息 ownership
- 与视觉合同的关键片段对齐

#### 2.2 权限守卫
前端单测已覆盖：
- 无权限访问时拦截
- 已登录访问登录页重定向
- 超管放行
- 未登录重定向登录页

#### 2.3 角色 / 用户 / 基础数据 / 仪表盘
前端单测已覆盖不少纯逻辑内容，例如：
- 角色按钮显隐逻辑
- 菜单树归一化
- 用户名和密码校验规则
- 用户启用/禁用逻辑
- 基础数据分类 / Tab / 关联字段
- 仪表盘卡片、待办、动态 mock 数据映射

### 3. 这一层的价值与局限

#### 3.1 价值
- 说明 Permission 模块并非只有 E2E
- 很多边界规则其实已经在单测层覆盖
- 对表单校验、按钮逻辑、静态映射、视觉结构漂移有一定保护作用

#### 3.2 局限
- 很多单测是在**重写组件逻辑**或**断言源码字符串存在**
- 这类测试适合防回归，但不等价于真实浏览器交互
- 它们无法替代：
  - 权限矩阵验证
  - 数据持久化验证
  - 接口前后状态变化验证
  - 真实 UI 流程验证

### 4. 对单测层的判断
- 当前前端单测**数量和覆盖面都不差**
- 但 fidelity 不均匀，存在不少“逻辑映射型”与“源码断言型”测试
- 所以这一层的结论是：**有覆盖，但不能单独证明业务流程正确**

## 第三层：后端单测与集成测试分析

### 1. 当前已有的后端测试资产

- `SysLoginControllerTest`
- `SysPasswordControllerTest`
- `SysPasswordControllerAnonymousIntegrationTest`
- `OsgDashboardControllerTest`
- `OsgBaseDataControllerTest`

### 2. 当前后端测试覆盖的内容

#### 2.1 登录与找回密码
后端已经覆盖了：
- 登录成功与 `rememberMe`
- `getInfo.firstLogin`
- 找回密码 3 个端点匿名访问
- `sendCode` 限流注解存在且参数正确

这意味着认证与找回密码并不是“只有前端测了”。

#### 2.2 仪表盘与基础数据
后端已覆盖：
- dashboard stats / todos / activities 的返回结构
- base data 列表、过滤、状态修改的成功/失败返回

### 3. 这一层仍然缺什么
- `REQ-PER-001` 权限矩阵没有被系统化转成后端授权测试
- 不同角色/不同数据范围的访问边界测试不足
- CRUD 之后的跨模块一致性验证不足
- 服务层 / 数据层真实协作的集成覆盖仍偏薄

### 4. 对后端层的判断
- 当前后端层**不是空白**
- 认证安全链路的覆盖明显强于业务后台 CRUD
- 问题不是“后端没测”，而是**授权语义与数据边界没有形成系统测试集**

## 第四层：门禁与框架能力分析

### 1. 当前框架已经做到的事

#### 1.1 `deliver-ticket` / `verification`
框架已经要求前端功能类 Ticket 必须给出 E2E 命令，不再允许只给截图或只给 build。

#### 1.2 `e2e-api-gate.sh`
当前门禁会：
- 统一入口运行 Permission E2E
- 先跑 `e2e_api_guard.py`
- 再跑 Playwright
- 检查代理 / 后端连接错误
- 输出行为契约报告路径

#### 1.3 `e2e_api_guard.py`
当前静态 guard 能发现一些低质量模式：
- `@api` 测试没有硬断言
- 使用软兜底 URL fallback
- `catch(() => false)` 这类弱模式
- `isVisible() + return` 的逃逸写法

### 2. 当前门禁没做到的事

#### 2.1 不知道“该测什么”
当前 guard 只能判断：
- 你有没有写测试
- 你有没有至少一条硬断言

它不能判断：
- 角色页是否必须包含“新增成功 + 删除拒绝 + 权限负例”
- 用户页是否必须包含“启用/禁用影响 + 角色生效”
- 仪表盘是否必须验证“角色数据过滤”

#### 2.2 不知道“需求是否已映射成测试”
当前门禁没有把这些真源接起来：
- `REQ-PER-001`
- 角色矩阵
- Ticket 的 AC
- E2E 用例清单

所以即使 E2E 命令能跑通，也不能证明关键需求已被转化为测试。

#### 2.3 不知道“覆盖是否完整”
当前没有任何机器可读清单来回答：
- 每个受保护页面是否至少有一个变更成功用例
- 是否至少有一个业务规则拒绝用例
- 是否至少有一个权限边界负例
- 是否至少有一个结果持久化断言

### 3. 对框架层的判断

当前框架解决的是：
- **强制执行测试**
- **阻止明显低质量的 E2E 写法**

当前框架尚未解决的是：
- **场景完整性**
- **需求到测试的可追踪性**
- **权限矩阵与数据边界的系统性覆盖**

## 一人公司开发框架的真实问题

基于以上四层证据，框架上的真实问题不是“测试缺失”，而是下面 4 点：

### 1. Source Stage 没有产出测试场景义务
Story / Ticket 阶段虽然已经能要求写 E2E，但还没有把以下义务机器化：
- 成功路径
- 业务规则拒绝路径
- 权限边界负例
- 持久化后置断言

### 2. Verification 只校验“存在”，不校验“完整”
当前验证更像是在问：
- 有没有跑测试
- 有没有硬断言

而不是在问：
- 是否覆盖了该页面最关键的语义

### 3. 权限矩阵没有转成测试清单
`REQ-PER-001` 这样的权限真源还停留在文档层，没有稳定映射成：
- 角色 × 页面 × 操作 × 数据范围
- 对应的正/负例用例清单

### 4. 测试层分工不清，导致误判
现在容易出现两种误判：
- 看到单测存在，就误以为业务 E2E 足够
- 看到 E2E 浅，就误以为整个模块没有测试

真实情况是：**有测试，但层次之间缺少分工契约和覆盖闭环。**

---

## 整改建议

### P0: 先把“覆盖义务”定义清楚
不要先喊 90% / 95% 这类当前无法自动验证的百分比门槛，先把每类页面的最低覆盖义务写成可执行规则。

建议对每个受保护后台页面至少要求：
- 1 个页面进入 / 基础加载用例
- 1 个变更成功用例
- 1 个业务规则拒绝用例
- 1 个权限或数据边界负例
- 1 个结果持久化或即时生效断言

### P0: 用需求真源驱动测试清单
对 Permission 模块，建议引入机器可读的测试覆盖清单，至少映射：
- 页面
- 关键操作
- 业务规则
- 角色/权限负例
- 对应用例文件

这样门禁才能检查“有没有漏掉整类场景”。

### P1: 优先补齐 4 个浅层页面的 E2E
优先级顺序建议：
1. 角色管理
2. 用户管理
3. 基础数据
4. 仪表盘

补齐时优先写：
- 真正变更数据的流程
- 失败路径
- 权限边界
- 结果断言

### P1: 保留并升级现有单测层
前端单测不应被否定，而应重新定位：
- 继续承担表单规则、纯逻辑、视觉结构漂移保护
- 对关键弹窗和关键表单，逐步增加更接近真实交互的组件级测试

### P2: 升级门禁，从“最低质量”走向“覆盖完整性”
可以分两步做：
- 第一步: 继续保留现有 `e2e_api_guard`，防止低质量写法回潮
- 第二步: 新增“覆盖清单校验”，检查每个关键页面的必要场景是否齐全

---

## 建议替代指标

与其写当前无法落地的百分比，不如先采用这些更可执行的指标：

### 页面级指标
- 每个后台页面至少 1 个成功变更用例
- 每个后台页面至少 1 个业务规则拒绝用例
- 每个后台页面至少 1 个权限/数据边界负例

### 认证安全指标
- 找回密码必须同时具备匿名访问、安全限流、token 生命周期、错误去重验证
- 登录必须具备成功、失败、登出、菜单可见性至少 4 类路径

### 框架指标
- 每个 Ticket 必须声明必测场景类别
- Verification 必须对照场景类别回读证据
- Gate 必须能检查“缺场景”而不只是“缺命令”

---

## 最终结论

Permission 模块当前的测试现状应被定义为：

- **不是无测试**
- **不是全层薄弱**
- **而是认证层相对深、CRUD 权限层偏浅、门禁层只到最低质量线**

所以，一人公司开发框架在测试覆盖上的真正问题是：

**它已经能强制“做测试”，但还不能稳定强制“做对测试、做全关键测试”。**

这也是下一轮框架整改应该瞄准的方向。
