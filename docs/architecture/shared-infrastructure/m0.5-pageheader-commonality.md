## 3. HTML / CSS 结构对比

### 3.1 现有 `<PageHeader>` 组件（Assistant + Admin 几乎一字不差）

**Template**：

```vue
<template>
  <header class="page-header">
    <div class="page-header__text">
      <h1 class="page-header__title">{{ title }}</h1>
      <p v-if="subtitle" class="page-header__subtitle">{{ subtitle }}</p>
      <p v-if="description" class="page-header__desc">{{ description }}</p>
    </div>
    <div v-if="$slots.actions" class="page-header__actions">
      <slot name="actions" />
    </div>
  </header>
</template>
```

**Props**：`title: string` / `subtitle?: string` / `description?: string`

**CSS（BEM）关键值**：

| 类 | 字号 | 颜色 | 特殊样式 |
|---|---|---|---|
| `.page-header__title` | 22px / 700 | var(--text, #1E293B) | — |
| `.page-header__subtitle` | 13px / 500 | var(--muted, #94A3B8) | **uppercase + letter-spacing 0.5px** |
| `.page-header__desc` | 14px | var(--text2, #64748B) | — |
| `.page-header__actions` | — | — | flex / gap 8 / flex-shrink 0 |

**唯一差异**：Assistant 多 `margin-bottom: 20px`，Admin 无。

### 3.2 双语机制（**关键发现**）

Assistant + Admin 现有组件**没有显式双语 prop**。实际用法是把英文塞进 `subtitle`：

```vue
<PageHeader title="学员求职总览" subtitle="Job Overview" description="查看我辅导和管理的学员求职进度">
```

视觉效果：
- title `学员求职总览`（22px 单独行）
- subtitle `JOB OVERVIEW`（13px **UPPERCASE** 灰色，**单独一行**，由 CSS `text-transform: uppercase` 强制大写）
- desc 副标说明（14px）

**与内嵌端的视觉差异**：
- 内嵌端：`学员求职总览 Job Overview`（中英**同一行**，英文紧贴中文，14px 灰色，**非 UPPERCASE**）
- 组件端：`学员求职总览` 占一行 / `JOB OVERVIEW` 单独一行 UPPERCASE
- 字号：内嵌 26-28px / 组件 22px

**这是非平凡的视觉差异**，不是改改 prop 名能解决。

### 3.4 双语标题机制（4 种变体实测分布）

| # | 写法 | 端 | view 数 |
|---|---|---|---|
| **V1** | `<h1 class="page-title">中 <span class="page-title-en">EN</span></h1>` | Mentor 5 + LM 7 + Asst legacy 3 + Student positions 1 | **16** |

| **V3** | `<h1 class="page-title"><span class="__zh">中</span><span class="__en">EN</span></h1>` | Student courses 1 | **1** |
| **V4** | `<PageHeader title="中" subtitle="EN" description="副">` | Asst 5 + Admin 30 | **35** |

> V1 是内嵌端最大公约数（45.7%）。V2 是 Student 本地约定（48.6%）。V4 是组件端承载机制（50%）。
> **全 5 端没有统一双语机制**。

## 4. 共性覆盖率计算（按路线图 §5.1.3）

### 4.1 公式与基础数据

每个 view 的 `.page-header` 块约 25 行（template 8-12 + CSS 15-30 平均）。

### 4.2 范围 A — 三端 SSOT（Asst + LM + Mentor，22 view）

#### 严格定义（仅看尚未 SSOT 化部分）

- 待抽 = Asst legacy 3 + LM 7 + Mentor 5 = 15 view
- 三端都用 V1 双语 + `.page-title / .page-title-en / .page-sub` → "同一份 shared 可覆盖"
- **共性覆盖率 ≈ 100%**（待抽 15 view 三端结构高度一致）

#### 综合（含已 SSOT + 待 SSOT）

- 已 SSOT 5（Asst V4）+ 待 SSOT 15 = 20 view 用同一 shared 后**完全等价**
- **综合共性 ≈ 95%**（与 M0.4 AppSidebar ~95% 一致）

### 4.3 范围 B — A + Student（42 view）

#### 严格

- Student 18/20 用 V2 裸 span / 28px / `OsgPageContainer` 嵌套 → 与三端 V1 + 26px 不同
- **共性覆盖率 ≈ 50-55%**

#### 放宽（双语机制选定 + 接受字号收敛）

- 双语机制只要选 1 个，Student 都能改造对齐（无业务变化）
- `OsgPageContainer` 嵌套不影响 PageHeader 抽取
- **共性覆盖率 ≈ 70-75%**

### 4.4 范围 C — A + Admin（70 view，含或不含 Student 两个子选项）

#### 严格

- Admin 30 已组件化 + Asst 5 已组件化 = 35 view 用同一组件（Asst + Admin 一字不差）
- **共性覆盖率 ≈ 60-65%**（含或不含 Student）

#### 放宽

- Admin + Asst 组件部分 100% → 直接接 shared
- **共性覆盖率 ≈ 75-80%**

### 4.5 共性覆盖率汇总

| 范围 | 严格 | 放宽 | 路线图判据 |
|---|---|---|---|
| **A 三端 SSOT（22 view）** | ~95%（待抽部分）/ ~95%（综合）| ~95% | ✅ ≥ 70% 标准抽取 |

## 7. 风险点 / 已知盲点

### 7.1 双语标题机制（最大风险）

四种现存写法 → 必须收敛 1 种。3 个候选：

#### 候选 P1：扩展 props（双 prop `titleZh + titleEn`）

```vue
<PageHeader :title-zh="学员求职总览" :title-en="Job Overview" :description="..." />
```

- ✅ 清晰 / 同行渲染 / 与 V1 视觉接近
- ❌ 与现有 `title + subtitle` 不兼容（35 已组件化 view 必须改 prop 名）

#### 候选 P2：保留 `title + subtitle`（最小改动）

```vue
<PageHeader title="学员求职总览" subtitle="Job Overview" description="..." />
```

- ✅ 35 已组件化 view 0 改动；接口零迁移成本
- ❌ subtitle 视觉是 UPPERCASE 单独行，与 V1 中英同行不一致；语义命名 misleading

#### 候选 P3：title 支持 slot（最灵活）

```vue
<PageHeader>
  <template #title>学员求职总览 <span class="title-en">Job Overview</span></template>
</PageHeader>
```

- ✅ 三种现有写法都可承载
- ❌ 失去类型约束 / 共性下降 / 违反 SSOT

> ⛔ 3 个方案都有 trade-off，留给主窗口拍板。Phase 1 方案文档全论证。

### 7.2 BEM vs 平铺 CSS 命名

| **平铺** | 改 35 + Admin 30 | 维持 | 改 |

### 7.3 Student `OsgPageContainer` 嵌套

### 7.4 Admin 端是否真要参与（路线图 §0.4 矛盾）

路线图 §0.4 明确"Admin 不对齐"，理由是"使用 ruoyi-ui (Vue2 + Element UI)"。

**实际事实**：
- `ruoyi-ui/`（Vue2）确实存在但**与 PageHeader 抽取无关**（不在 osg-frontend 下）
- `osg-frontend/packages/admin/` 是 **Vue3 + Ant Design Vue**（与三端 SSOT 同栈）
- Admin 端 `osg-frontend/packages/admin/src/components/PageHeader.vue` Vue3 BEM 组件**与 Assistant 一字不差**
- 30 view 已用 `<PageHeader>`（与 Assistant 模式一致）

**结论**：
- 路线图 §0.4 的"Admin 不对齐"是针对 ruoyi-ui，**不是** osg-frontend/admin
- osg-frontend/admin 在 PageHeader case 下**实质上已经与 Assistant SSOT 同步**
- 范围 C 不是"违反路线图"，而是"§0.4 没说清楚 Vue3 admin 与 Vue2 ruoyi-ui 是不同 scope"

⚠️ **主窗口需要：选 C 之前先决定路线图 §0.4 是否要补一句澄清**。
