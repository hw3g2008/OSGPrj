## 1. 背景与决策依据

### 1.1 路线图判据复述

路线图 §4.1.2 降级判据（M0.4 同款）：

| 共性覆盖率 | 决策 |
|---|---|
| ≥ 70% | ✅ 全抽 shared 组件 |
| 50% - 70% | ⚠️ 抽 BaseXxx + 各端包 Xxx |
| < 50% | ❌ 降级为"只抽个别原子" |

Phase 0 实测共性覆盖率（`m0.5-pageheader-commonality.md` §4.5）：

| 范围 | 严格 | 放宽 | 结论 |
|---|---|---|---|
| A 三端 SSOT（22 view）| ~95% | ~95% | ✅ 标准抽取 |
| B + Student（42 view）| ~50-55% | ~70-75% | ⚠️ 临界 |
| C + Admin（70 view）| ~60-65% | ~75-80% | ✅ 标准抽取 |

### 1.2 与 M0.4 的本质差异

| 维度 | M0.4 MainLayout | M0.5 PageHeader |
|---|---|---|
| 组件粒度 | 大（壳）| 小（原子展示）|
| 已存在 shared 候选？ | ❌ 不存在 | ✅ Asst + Admin 已各 fork 一份（一字不差，仅 `margin-bottom` 差异）|
| 共性瓶颈 | 端特有黑洞 ~35%（navigationGroups / badge / toast / roleLabel）| **无端特有业务**，差异仅在视觉语法 + 双语机制 |
| 决策核心 | 抽 vs 不抽 | **范围 + 双语机制 + CSS 命名** |

**M0.4 的关键启示**：
- 子组件层共性 ~95%，远高于整壳 18.7%
- 抽子组件后 `m0.4-step5-pr-summary` 实测三端代码减重 51.6%

**M0.5 的事实**：
- PageHeader **本身已是子组件层**，默认就在高共性区
- 共性瓶颈不是"端特有黑洞"，而是**4 种双语写法 + 26/28/22 px 字号 + BEM/平铺类名**这些**纯视觉语法分歧**

### 1.3 推荐方向（Phase 0 §8 已论证）

**Phase 0 倾向以下方向，**最终拍板权在主窗口****：

| 决策点 | 推荐 | 备选 | 配套约束 |
|---|---|---|---|
| **D-Range** | C-1（A + Admin，**不含 Student**）| A（最小）/ C（含 Student，最大）| — |
| **D-Bilingual** | P1（双 prop `titleZh + titleEn`）| P2（保留 `title + subtitle`）/ P3（slot）| ⚠️ P1 + D-Range = A 不一致：Admin 30 view 不改造则留两套 prop 接口，建议范围 A 配套选 P2（详见 §10.1）|
| **D-Naming** | BEM（与 M0.4 AppSidebar 一致）| 平铺 | — |
| **D-StudentNesting** | 选 C-1 → N/A（已 grep 验证：LM PageHeader 7 view 与 OsgPageContainer 6 view 无重叠，详见 §7.1 R4）| 选 B/C → 必须额外评估 `OsgPageContainer` 嵌套间距 | — |
| **D-AdminScope** | 路线图 §0.4 补一句澄清 | 维持原文 | — |

**本方案文档按推荐方向（C-1 + P1 + BEM）展开**。如主窗口拍板选其他组合，§3-§9 标注了"加减点"。

## 2. 范围决策矩阵（⛔ 主窗口必须拍板）

### 2.1 三种范围对比（详见 Phase 0 §6）

| # | 范围 | view 覆盖 | 严格 / 放宽 | 工时 | 路线图严格度 |
|---|---|---|---|---|---|
| **A** | Asst + LM + Mentor 三端 SSOT | 22 | ~95% / ~95% | ~6h | ✅ 严格遵循 §0.4 |
| **C-1**（推荐）| A + osg-frontend/admin Vue3 | 50 | ~60-65% / ~75-80% | ~9-10h | ⚠️ 路线图 §0.4 需澄清 |
| **C** | A + Admin + Student | 70 | ~60-65% / ~75-80% | ~12-15h | ⚠️ 路线图 §0.4 需澄清 + Student 嵌套需评估 |

### 2.2 D-AdminScope 子决策（**选 C-1 / C 必读**）

路线图 §0.4 原文："Admin 不在 SSOT 对齐范围（Vue2 ruoyi-ui + Element UI 不兼容）"。

**Phase 0 §7.4 实测事实**：

| 端 | 技术栈 | PageHeader 实现 | 与 Asst SSOT 的关系 |
|---|---|---|---|
| `ruoyi-ui/`（Vue2 老前端）| Vue2 + Element UI | 无 PageHeader 组件 | 真正不兼容（与本 case 无关）|
| `osg-frontend/packages/admin`（Vue3 新前端）| Vue3 + Ant Design Vue | 有 `PageHeader.vue`（与 Asst 一字不差）| **实质上已 fork SSOT**，只是物理上没共享 |

**结论**：`osg-frontend/packages/admin` 不是 §0.4 排除的对象。范围 C-1 / C 不是违反路线图，而是**§0.4 没说清楚 Vue3 admin 与 Vue2 ruoyi-ui 是不同 scope**。

**主窗口决策项**：
- **选项 1**：路线图 §0.4 补一句澄清（"不对齐范围仅指 ruoyi-ui Vue2；osg-frontend/admin Vue3 各 case 评估"），范围 C-1 / C 合规
- **选项 2**：维持 §0.4 原文，本期范围限定 A，admin/PageHeader.vue 留待未来
- **选项 3**：直接修订 §0.4，明确 osg-frontend/admin Vue3 入 SSOT 范围

### 2.3 D-Range 子决策

| 子决策 | 选项 | 理由 |
|---|---|---|
| **是否含 Student** | 推荐**不含**（C-1）| Student 20 view 套 OsgPageContainer，需评估嵌套关系 + 双语机制 V2 裸 span 需统一改造，工作量翻倍 |
| **是否含 Admin** | 推荐**含**（C-1）| 删 admin/PageHeader.vue 65 行重复，避免认知债 |

> 主窗口若坚持范围 A，§3-§9 仍按 BEM + P1 写（推荐），但接入 view 列表减少 30 个 Admin。

## 3. 抽取范围与接口设计

> 以下按推荐方向 **C-1 + P1 + BEM** 展开。如选 A，去掉 Admin 30 view 接入；如选不同双语，§3.2 改 prop 命名。

### 3.1 抽入 `@osg/shared/components/PageHeader/`

```
@osg/shared/components/PageHeader/
├── PageHeader.vue       # 主组件（template + script + style）
├── types.ts             # PageHeaderProps 接口
├── index.ts             # re-export
└── __tests__/
    └── PageHeader.spec.ts  # unit tests
```

**Template 抽入**：
- `<header class="page-header">` 根容器
- `<div class="page-header__text">` 文本块（v-if 主标 / 副标 / 描述）
- `<h1 class="page-header__title">` 中文主标
- `<span class="page-header__title-en">` 英文标题（**同行内联**）
- `<p class="page-header__desc">` 副标说明
- `<div class="page-header__actions">` 右侧 actions（slot 包裹）

**Script 抽入**：
- 仅 props 定义（PageHeader 是纯展示原子，无内部状态 / 事件 / lifecycle）

**Style 抽入**：
- `.page-header` 容器布局
- `.page-header__title` 26px 主标
- `.page-header__title-en` 14px 内联灰色英文（**同行**渲染，与 V1 视觉一致）
- `.page-header__desc` 14px 副标
- `.page-header__actions` 右侧 actions 容器

### 3.2 Props（D-Bilingual = P1 双 prop）

```typescript
interface PageHeaderProps {
  // === 必传 ===
  titleZh: string         // 中文主标，例 '学员求职总览'

  // === 可选 ===
  titleEn?: string        // 英文标题，例 'Job Overview'，同行内联渲染
  description?: string    // 副标说明，例 '查看我辅导和管理的学员求职进度'
}
```

**对比方案 P2（保留 `title + subtitle`）**：

| 维度 | P1（推荐）| P2（备选）|
|---|---|---|
| Prop 命名 | titleZh / titleEn | title / subtitle |
| 35 已组件化 view 改动 | **必须改 prop 名**（subtitle → titleEn）| 无改动 |
| 视觉与内嵌端 V1 对齐 | ✅ 同行内联（接近 V1 现状）| ❌ 单独行 UPPERCASE（与 V1 不一致）|
| 语义清晰度 | ✅ titleEn 即英文标题 | ❌ subtitle 实际承担英文标题角色，misleading |
| 字号决策 | 主标 26px / 英文 14px（与 V1 一致）| 现有 22px / 13px UPPERCASE |

**对比方案 P3（slot 化）**：

| 维度 | P1 | P3 |
|---|---|---|
| 类型约束 | ✅ 强类型 | ❌ slot 内 HTML 自由 |
| 共性 | ✅ 各 view 接口一致 | ❌ 每个 view 自己写双语 HTML，反 SSOT |
| 灵活性 | ⚠️ 受限（仅支持中英两段）| ✅ 可承载任意结构 |

> P3 不推荐，违反路线图原则 D（数据结构统一）+ 原则 E（YAGNI 命名清晰）。

### 3.3 actions Slot

```typescript
interface PageHeaderSlots {
  actions?: () => unknown    // 右侧操作区（按钮 / pill / dropdown 等）
}
```

**用法**：

```vue
<PageHeader title-zh="学员求职总览" title-en="Job Overview" description="查看我辅导和管理的学员求职进度">
  <template #actions>
    <a-button @click="handleExport">
      <template #icon><ExportOutlined /></template>
      导出
    </a-button>
  </template>
</PageHeader>
```

actions 区可承载现有 4 种现状：
- A1（slot）：直接放 `<template #actions>` ← 已组件化 view 维持
- A2（显式 div）：删 `.page-header__actions` div，挪入 slot
- A3（平铺兄弟元素）：button / a-button 挪入 slot
- A4（无 actions）：不写 slot

### 3.4 默认值与边界

| Prop / Slot | 缺省行为 |
|---|---|
| `titleEn` 未传 | 不渲染 `<span class="page-header__title-en">`（v-if） |
| `description` 未传 | 不渲染 `<p class="page-header__desc">`（v-if）|
| `actions` slot 无内容 | 不渲染 `<div class="page-header__actions">`（v-if `$slots.actions`）|
| `titleZh` 必传 | 编译期类型检查报错（必须传）|

## 4. 内部行为

PageHeader 是纯展示原子组件，**无内部状态 / 无 emit / 无 lifecycle**。

唯一的"行为"是按 props 渲染 DOM。

> **`/validate-doc` Round 1 假设 1 翻转**：检查"是否真的纯展示"——实测 Asst + Admin 现有 PageHeader.vue 仅有 `defineProps`，无 `defineEmits` / 无 lifecycle / 无响应式 ref，✅ 假设成立。

### 4.1 视觉规则（CSS 决定）

```css
.page-header {
  display: flex;
  align-items: flex-start;
  justify-content: space-between;
  gap: 16px;
  margin-bottom: 24px;          /* §4.2 决策：选 24px */
}

.page-header__text {
  display: flex;
  flex-direction: column;
  gap: 6px;
}

.page-header__title {
  margin: 0;
  font-size: 26px;
  font-weight: 700;
  color: var(--text, #1E293B);
  line-height: 1.3;
}

.page-header__title-en {
  margin-left: 8px;
  font-size: 14px;
  font-weight: 400;
  color: var(--muted, #94A3B8);
  /* 关键：不强制 uppercase，与 V1 内嵌端视觉一致 */
}

.page-header__desc {
  margin: 0;
  font-size: 14px;
  color: var(--text2, #64748B);
}

.page-header__actions {
  display: flex;
  align-items: center;
  gap: 8px;
  flex-shrink: 0;
}

@media (max-width: 768px) {
  .page-header {
    flex-direction: column;
    align-items: stretch;
  }
}
```

### 4.2 `margin-bottom` 决策

Phase 0 §7.5 列出 3 个候选：

| 选项 | 现有 view 受影响 | 决策 |
|---|---|---|
| 0（无）| Asst 5 + Admin 30 共 35 view 多了 24px 间距（视觉变紧凑）| ❌ 破坏现状 |
| 20px（Asst 现状）| Admin 30 多 20px / 内嵌端 35 少 4px | ⚠️ 折中但视觉敏感 |
| **24px（推荐）**| Asst 5 + Admin 30 多 4px（极小）/ 内嵌端 35 维持 | ✅ 最小破坏 |
| 暴露 prop | 接口膨胀，反 SSOT | ❌ |

**推荐 24px**：与内嵌端（Mentor / LM / Student）现状一致，对 Asst 5 + Admin 30 仅多 4px（视觉容差内）。

**`/validate-doc` Round 1 假设 7 翻转 — grep 实测结果**：

```
Mentor 5 view + PlaceholderPage = 6/6 用 24px ✅
Asst legacy 3 view: schedule = 24px / AssistantPlaceholderShell = 24px / profile = 20px ⚠️
LM 7 view: 全部 24px ✅
```

**结论**：`Asst legacy profile` 唯一异类（20px），其他 14 view 都是 24px。统一为 24px 后 `profile` 多 4px 间距，视觉容差内。

### 4.3 `<header>` vs `<div>` 语义化

现有 Asst + Admin 组件用 `<header>`，Mentor/LM/Student 内嵌用 `<div>`。

抽 shared 后**统一为 `<header>`**（语义化标签更准确，符合 a11y 最佳实践）。
