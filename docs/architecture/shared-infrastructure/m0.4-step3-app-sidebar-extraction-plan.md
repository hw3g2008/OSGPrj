## 1. 背景与决策依据

### 1.1 为什么不抽整壳 MainLayout

路线图 §4.1.2 的降级判据：

| 共性覆盖率 | 决策 |
|---|---|
| ≥ 70% | ✅ 全抽 `<MainLayout>` 到 shared |
| 50% - 70% | ⚠️ 抽 `<BaseLayout>` + 各端包 `<MainLayout>` |
| < 50% | ❌ 降级为"只抽 PageHeader" |

Round 2 commonality 报告结果：**45.8%**（临界值以下）。

三个"端特有黑洞"决定了整壳抽取不划算：

| 黑洞 | 占比 | 性质 |
|---|---|---|
| `navigationGroups` 数据 | ~24%/端 | 业务数据，各端路径/标签不同，不可共用 |
| 端特有 logic（badge/toast/roleLabel）| ~8%/端 | 按路线图原则 D"端特有业务永远在端内" |
| 接口分歧（LM 拦截 navigate vs Asst/Mentor 直跳）| ~3% | 若统一需改 LM 业务行为 |

**整壳抽取的理论上限 ~65-70%**，45.8% 已经是"接近天花板"的水平。

### 1.2 为什么抽子组件 AppSidebar

从 commonality 报告 §3.1-3.3 可见：
- Sidebar **template + style** 在 Round 2 中几乎 100% 一致
- 差异仅在 `navigationGroups`（数据）和 `logout Modal.confirm`（Step 1.5 已统一）

**子组件层共性 ~95%**，远高于整壳的 45.8%。

决策：**方案 B1 — 抽 AppSidebar 子组件**。

### 1.3 决策的杠杆效应

| 前置投入 | 本方案收益 |
|---|---|
| Step 1 — Mentor 对齐 Asst/LM 风格（~260 行重构）| Sidebar template/style 三端一致 → 可抽 |
| Step 1.5 — 三端 logout 升级 Modal.confirm（~60 行）| Sidebar logout 行为一致 + 遵守 ant-design-vue UI 库规范 |
| **合计投入** ~320 行 | **杠杆**：抽 AppSidebar 后三端 sidebar 一次维护，消除 ~420 行重复 |

## 3. Props / Emits 接口设计

### 3.1 Props

```typescript
interface AppSidebarProps {
  // === 必传 ===
  // 菜单数据（各端不同）
  navigationGroups: NavigationGroup[]
  // 用户信息
  displayName: string        // 如 'Amy' / 'Jess (Lead Mentor)' / 'Matt (Mentor)'
  userInitials: string       // 如 'AM' / 'JE' / 'ME'
  roleLabel: string          // 如 '学员助理' / '培训主管' / '导师'
  // 当前路由（用于 isActive）
  currentPath: string        // 通常来自 useRoute().fullPath

  // === 可选（有默认值）===
  logoTitle?: string         // 默认 'OSG'
  logoSubtitle?: string      // 默认 '就业指导 Employment'
  userId?: string | number   // 显示在 user-card 上（可选）
  homePath?: string          // 默认 '/home'
  profilePath?: string       // 默认 '/profile'
}
```

### 3.2 NavigationGroup / NavigationItem

```typescript
interface NavigationGroup {
  title: string              // 如 '求职辅导 Career' / '教学管理 Teaching'
  items: NavigationItem[]
}

interface NavigationItem {
  path: string               // 路由路径，如 '/career/positions'
  label: string              // 显示文字，如 '岗位库 Positions'
  iconClass: string          // 图标 class，如 'mdi mdi-briefcase-outline'
  activePaths: string[]      // 高亮触发前缀列表
  badge?: number             // 可选徽章数字。仅在 badge > 0 时渲染数字徽章；undefined / 0 不显示
  hidden?: boolean           // 可选隐藏。true=该 item 根本不渲染（DOM 不存在）；
                             // 注："敬请期待"等"显示+点击拦截"的场景不要用 hidden，
                             // 而应在端的 nav emit handler 内拦截（参考 LM 的 AVAILABLE_NAVIGATION_PATHS）
}
```

### 3.3 Emits

```typescript
interface AppSidebarEmits {
  // 点击导航项（路径作为 payload，端可拦截）
  (e: 'nav', path: string): void
  // 点击个人中心入口
  (e: 'profile-click'): void
  // 用户确认退出登录（Modal.confirm 已在 AppSidebar 内部完成）
  (e: 'logout'): void
}
```

**设计要点**：
- `nav` emit 把决策权交给端，LM 可以拦截到 `handleUpcomingNavigation`，Asst/Mentor 可以直接 `router.push`
- `logout` emit 代表"用户已在 Modal 上点了确定"，端只需执行清 auth + 跳转
- 不用 slots（所有差异都通过 props/emits 覆盖，接口更干净）

**emit 语义对称性说明**（避免读者疑惑"为什么 logout 不和 nav 对齐"）：

| Emit | 语义类型 | 触发时机 | 端的责任 |
|---|---|---|---|
| `nav` | **触发型**（用户点击）| 用户点击 nav-item | 端决策"如何处理"（router.push 或 message.info 拦截）|
| `profile-click` | **触发型**（用户点击）| 用户点击"个人设置" | 端决策跳转目标（不同端 profilePath 不同）|
| `logout` | **确认型**（用户已确认）| 用户在 Modal 上点击"确定" | 端只需执行副作用（clearAuth + router.push('/login')）|

两类 emit 语义都合理：触发型让端**决定行为**；确认型让端**承担副作用**。`logout` 的 `Modal.confirm` 在 sidebar 内是为了 DRY（避免三端各自写一遍 Modal 配置），不破坏端的自主权——端仍然可以通过 `@logout=` 拦截/扩展逻辑（如 logout 前埋点、logout 后跳到不同登录页）。

## 4. 内部行为

### 4.1 `showUserMenu` 管理

```typescript
const showUserMenu = ref(false)

function toggleUserMenu() {
  showUserMenu.value = !showUserMenu.value
}

function closeUserMenu() {
  showUserMenu.value = false
}

// document click listener
onMounted(() => {
  document.addEventListener('click', handleOutsideClick)
})
onBeforeUnmount(() => {
  document.removeEventListener('click', handleOutsideClick)
})

function handleOutsideClick(event: MouseEvent) {
  const target = event.target as HTMLElement
  if (!target.closest('.sidebar-footer')) {
    closeUserMenu()
  }
}

// ESC key listener
onMounted(() => {
  window.addEventListener('keydown', handleEscKey)
})
onBeforeUnmount(() => {
  window.removeEventListener('keydown', handleEscKey)
})

function handleEscKey(event: KeyboardEvent) {
  if (event.key === 'Escape') closeUserMenu()
}
```

### 4.2 `handleNavClick` — 内部 emit 而非直接跳转

```typescript
function handleNavClick(item: NavigationItem) {
  if (item.hidden) return
  emit('nav', item.path)
  // 菜单项点击不关闭 user-menu（user-menu 只在 user-card 交互区域）
}
```

**为什么不在内部 router.push**：
- LM 需要拦截未实现路由 → 弹"敬请期待" toast
- 若 sidebar 直接跳转，LM 的拦截无处可做
- emit 模式把决策权下放，各端自主实现

### 4.3 `handleProfileClick` / `handleLogoutClick`

```typescript
function handleProfileClick() {
  closeUserMenu()
  emit('profile-click')
}

function handleLogoutClick() {
  closeUserMenu()
  Modal.confirm({
    title: '确定要退出登录吗？',
    okText: '确定',
    cancelText: '取消',
    onOk: () => emit('logout'),
  })
}
```

### 4.4 `isActive` 计算

```typescript
function isActive(item: NavigationItem): boolean {
  return item.activePaths.some((prefix) => props.currentPath.startsWith(prefix))
}
```

由端传入 `currentPath` prop（通常来自 `useRoute().fullPath`），sidebar 不依赖 vue-router（减少耦合，便于测试）。

```vue
<script setup lang="ts">
import { computed, ref } from 'vue'
import { useRoute, useRouter } from 'vue-router'
import { AppSidebar, type NavigationGroup } from '@osg/shared/components/AppSidebar'
import { clearAuth, getUser } from '@osg/shared/utils'

const route = useRoute()
const router = useRouter()

const navigationGroups = computed<NavigationGroup[]>(() => [
  /* ... Assistant 特有的 6 个分组 ... */
])

const displayName = computed(() => getUser()?.nickName ?? 'Assistant')
const userInitials = computed(() => 'AM')

// Assistant 特有：roleLabel 复杂计算
const roleLabel = computed(() => {
  const roles = getUser()?.roles ?? []
  if (!Array.isArray(roles) || roles.length === 0) return '学员助理'
  // 支持 ['assistant', 'admin'] / [{name: 'assistant'}, ...] 等多形态
  const firstRole = roles[0]
  if (typeof firstRole === 'string') {
    return roleMap[firstRole] ?? '学员助理'
  }
  if (firstRole && typeof firstRole === 'object' && 'name' in firstRole) {
    return roleMap[(firstRole as { name: string }).name] ?? '学员助理'
  }
  return '学员助理'
})

const roleMap: Record<string, string> = {
  assistant: '学员助理',
  admin: '管理员',
}

function handleNavClick(path: string) {
  router.push(path)
}

function handleProfileClick() {
  router.push('/profile')
}

function handleLogout() {
  clearAuth()
  void router.push('/login')
}
</script>
```
