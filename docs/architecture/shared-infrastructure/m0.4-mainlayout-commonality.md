### 2.1 三端结构骨架（高层一览）

三端都采用同一套"左侧 sidebar + 右侧 main" 布局骨架：

```
<div class="{main-shell | app-layout}">   ← 根类名 Asst/LM vs Mentor 差异
  <aside class="sidebar">
    <div class="sidebar-header"> logo-icon + logo-text </div>
    <nav class="sidebar-nav">
      <!-- 首页占位（v-show="false"）-->
      <!-- 菜单项（Asst/LM 数据驱动 v-for / Mentor 硬编码）-->
    </nav>
    <div class="sidebar-footer">
      <button/div class="user-card"> 头像 + 姓名 </...>
      <div v-if="showUserMenu" class="user-menu"> 个人设置 / 退出登录 </div>
    </div>
  </aside>
  <main class="{main | main-content}">   ← main 类名差异
    <router-view />
  </main>
</div>
```

**共同视觉规则**（设计系统层面）：
- Sidebar 固定宽 260px，位置 fixed
- Primary color `#7399C6`，primary-gradient `linear-gradient(135deg, #7399C6 0%, #9BB8D9 100%)`
- 用户头像 40×40px 圆形 + 渐变背景
- 菜单项 hover/active 色彩一致

## 3. 行级共性分析

### 3.1 Template 段

| 项目 | Assistant 实现 | Lead-Mentor 实现 | Mentor 实现 | 可共用? |
|---|---|---|---|---|
| 根类名 + 布局 shell | `.main-shell` | `.main-shell` | `.app-layout` ⚠️ | ⚠️（需统一类名）|
| sidebar-header + logo | 8 行 | 6 行（紧凑）| 6 行 | ✅ 结构共用，内容各端传入 |
| nav wrap | 2 行 | 2 行 | 2 行 | ✅ |
| 首页占位 `v-show="false"` | 11 行（数据驱动）| 10 行（数据驱动）| 3 行（硬编码）| ⚠️ |
| 菜单循环（核心）| `v-for navigationGroups` 16 行 | `v-for navigationGroups` 16 行 | **硬编码 ~40 行**（每项独立 `<a>`）| ❌ **不可共用**（实现方式不同）|
| sidebar-footer + user-card | 7 行（`<button>`）| 7 行（`<button>`）| 7 行（`<div>` ⚠️）| ✅ 结构共用 |
| user-menu | 11 行（"个人设置 Profile" + "退出登录 Logout"）| 8 行（"个人设置" + "退出登录"）| 8 行（同 LM）| ✅ 结构共用 |
| main + router-view | 3 行 | 3 行 | 7 行（多 `<transition>`）| ⚠️（Mentor 额外包装）|

**严格共用行数**（"同一份 shared 实现可覆盖"）：
- 骨架：根 div + aside + main 壳 3 行
- sidebar-header 结构 6 行
- nav wrap 2 行
- sidebar-footer + user-card + user-menu 结构 15 行
- main + router-view 2 行
- **小计：~28 行**

**不可共用**：
- 菜单循环（Asst/LM 数据驱动 vs Mentor 硬编码，需二选一实现方式）
- Mentor 的 `<transition>` 包装
- 类名（`.main-shell` vs `.app-layout`；`.main` vs `.main-content`）

### 3.3 Style 段

#### Assistant vs Lead-Mentor

- 都使用 `scoped lang="scss"` + CSS variables (`--primary`, `--primary-light`, `--primary-gradient`, `--text`, `--text2`, `--muted`, `--border`, `--bg`)
- 类名完全一致：`.main-shell` / `.sidebar` / `.sidebar-header` / `.sidebar-logo` / `.logo-icon` / `.logo-text` / `.sidebar-nav` / `.nav-section` / `.nav-item` / `.nav-badge` / `.sidebar-footer` / `.user-card` / `.user-avatar` / `.user-info` / `.user-menu` / `.user-menu-item` / `.main`
- **代码相似度 ~92%**
- 仅差异：
  - LM 多一个 `--card-shadow` 变量定义（未使用）
  - LM `user-card` 背景 `var(--primary-light)`，Assistant 是 `#fff`
  - LM 多一个 `@media (max-width: 1100px)` sticky 断点

#### Mentor vs Asst/LM

- ❌ **纯 CSS**（无 SCSS）
- ❌ **硬编码颜色值**（无 CSS variables）— 如 `#E8F0F8` / `#64748B` / `#7399C6`
- ❌ **类名不一致**：`.app-layout`（非 `.main-shell`）、`.main-content`（非 `.main`）
- ❌ **全部样式压缩为单行写法**
- ❌ 无响应式 `@media`
- ❌ 多出 transition 样式 `.fade-enter-active` / `.fade-enter-from` / `.fade-leave-to`
- ✅ 视觉规则等价（色值、尺寸、间距几乎一致）

**严格共用行数**（"同一份实现可覆盖"）：
- 视觉规则相同但代码实现不同 ≠ 共用
- Mentor 的类名与 Asst/LM 无交集
- **小计：~30 行**（仅"如果三端重构统一，可共用的最小基础集"）

#### 放宽视角（假设 Mentor 重构为 SCSS + 对齐类名 + 使用 CSS variables）

- CSS variables 10 行 + 样式规则 ~180 行可共用
- 潜在共性 ~190 行

## 4. 共性覆盖率计算（按 §5.1.3）

### 4.1 公式

```
总代码行数 = Σ(三端行数) / 3 = (525 + 522 + 237) / 3 = 428 行
共性覆盖率 = 总共性行数 / 总代码行数
```

### 4.2 方式 A（严格 — §5.1.3 原意："仅视觉相似但逻辑不同的不算共用"）

| 段 | 共性行数 |
|---|---|
| Template | 28 |
| Script | 22 |
| Style | 30 |
| **合计** | **80** |

**共性覆盖率 = 80 / 428 ≈ 18.7%**

### 4.3 方式 B（放宽 — 假设先把 Mentor 重构到与 Asst/LM 对齐后再衡量）

| 段 | 共性行数 |
|---|---|
| Template | ~49 |
| Script | ~47 |
| Style | ~190 |
| **合计** | **~286** |

**共性覆盖率 = 286 / 428 ≈ 66.8%**

### 4.4 采用哪个数字？

路线图 §5.1.3 明确：

> **"共用"定义**：三端代码逻辑可以用同一份 shared 实现覆盖的部分。**仅视觉相似但逻辑不同的不算共用**。

Mentor 的 style 视觉与 Asst/LM 高度相似，但代码实现方式完全不同（纯 CSS + 硬编码 + 类名差异），**按严格定义不算共用**。

**→ 应采用方式 A：共性覆盖率 ≈ 18.7%**

### 4.5 补充观察（非决策依据）

若只统计 **Assistant 和 Lead-Mentor 两端**（排除 Mentor）：

| 段 | 两端共性行数 |
|---|---|
| Template | ~52 |
| Script | ~85（含 interface + filteredNavigationGroups + 共用函数）|
| Style | ~200 |
| **合计** | **~337** |

两端平均 523.5 行 → **Asst/LM 两端共性覆盖率 ≈ 64%**

> 说明：Mentor 是三端共性的最大 outlier，拖低三端平均。Assistant/LM 两端已近乎 fork 关系。

## 5. 决策建议

### 5.1 严格按路线图判据（方式 A）

- 共性覆盖率 **18.7% < 50%**
- 触发路线图 §1.6.1 **D5 升级触发器**："三端 MainLayout 结构差异 >50%"
- 对照 §4.1.2 降级判据：**降级为"只抽 PageHeader，MainLayout 各端保留"**

### 5.2 三种候选方案对比

| # | 方案 | 工作量 | 收益 | 风险 | 推荐度 |
|---|---|---|---|---|---|
| **C（严格降级）** | M0.4 跳过 MainLayout 抽取，M0.5 只抽 `<PageHeader>`（M0.4 / M0.5 合并）| 小 | 最小抽取，遵循 YAGNI | 三端 MainLayout 继续漂移；未来 Mentor 重构时需重新评估 | ✅✅✅ **强推荐** |
| B（先重构 Mentor → 再抽 BaseLayout）| M0.4 先把 Mentor 从硬编码改为数据驱动 + SCSS，然后抽 `<BaseLayout>` | **大**（Mentor 重构 ~100 行 + BaseLayout 设计）| 三端 MainLayout 结构对齐 | 超出 M0 范围；Mentor badge 动态刷新属端特有业务，抽完仍需各端包装 | ⚠️ 不推荐 |
| A（强抽 `<MainLayout>` to shared）| 抽整个 MainLayout 到 shared，用 props / slots 覆盖差异 | **极大** | — | 违反原则 D "端特有业务永远在端内"；slot/prop 爆炸 | ❌ 不推荐 |
| D（Asst/LM 两端先抽 + Mentor 延后）| 走 §7.5 两端降级；Asst/LM 共用 MainLayout，Mentor 保留 | 中 | Asst/LM 消除 ~60% 重复 | 违反三端对齐原则；Mentor navigationGroups 路径结构本来就与 LM 不同，即使抽壳也要两端传不同数据 | ⚠️ 备选 |

### 5.3 推荐方案：**C（降级只抽 PageHeader）**

**理由**：

1. **严格遵循路线图判据**（18.7% 远低于 50% 门槛）
2. **最短路径 / YAGNI**（符合用户"不考虑效率只考虑质量"原则的"简约不等于省略" — 简单路径能达成 M0 目标就不绕弯）
3. **尊重 Mentor 技术债现状**（硬编码 + 纯 CSS 是 Mentor 端的历史遗留，独立处理比塞进 shared 抽取更清晰）
4. **下游影响可控**：M1-M5 子 Epic 不依赖 `<MainLayout>` shared（路线图 §4.0.2 "M0.4-M0.6 是 M1-M5 前端的**软依赖**"）
5. **可逆性保留**：将来若 Mentor 完成独立重构（数据驱动化 + SCSS 化 + 类名对齐），三端共性会自动涨到 ~70%，届时可重启 M0.4 抽取（符合 §5.1.2 推迟后的再评估机制）

**M0.4 执行动作**（方案 C 采纳后）：

1. M0.4 标记为 **降级完成**（不抽 MainLayout）
2. M0.5 `PageHeader` 抽取合并进来，作为 M0 前端基础设施的唯一交付物（M0.4 / M0.5 合并为单个节点）
3. 把"Mentor MainLayout 数据驱动化重构"登记为独立技术债（可归档到 `m0-violation-audit.md` 或新开 M0.7+ 节点），不在本 M0 范围内
4. 更新路线图 §4.1 M0.4 状态为 "✅ 已降级（共性覆盖率 18.7%）"，并在进度追踪表中标注降级原因
5. 更新路线图 §6.1 顶层追踪

## 8. 附录：行数粗略估算方法说明

- 每段（Template / Script / Style）按语义块（组件 / 函数 / 选择器组）分块计数
- "共性行数"取保守估算：三端都有且"同一份 shared 实现可完整覆盖"的代码块
- 端特有业务（如 Mentor 动态 badge、LM 敬请期待 toast、Assistant 复杂 roleLabel）**不计入共性**
- 数据内容（如 navigationGroups 的具体路径）**不计入共性**（各端传入）
- 仅计入：结构 / 类型定义 / 通用逻辑函数 / 基础选择器结构

**量化口径与 §5.1.3 严格一致：不计"视觉相似但代码实现不同"**。
