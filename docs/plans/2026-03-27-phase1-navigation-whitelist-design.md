# OSG 第一期导航白名单设计稿

日期：2026-03-27  
主题：按“系统第一期需求大纲”隐藏 5 端未落地菜单入口，仅保留第一期已落地功能的导航可见性  
状态：设计已确认，待进入实现

## 1. 背景

当前 5 个端的前端代码已经包含了大量二期或未正式开放的页面入口，主要表现为：

- 侧边栏中存在超出第一期交付范围的菜单项
- 多个端默认登录后会落到首页，而首页本身不属于本次希望暴露给用户的正式入口
- 各端路由已经存在 `placeholder`、`coming-soon`、半成品页面和历史保留页面，用户从正常导航路径能看到过多未落地功能

本次目标不是裁撤路由或做权限体系重构，而是严格按照第一期已落地范围，收口“用户正常浏览时能看到什么”。

## 2. 目标

本次目标固定为三件事：

1. 5 个端的侧边栏只展示第一期已落地功能。
2. 首页本身从正常导航路径中移除，用户登录后不再默认落到首页。
3. 登录、忘记密码以及手动输入 URL 的现有行为全部保持不变。

## 3. 非目标

本次明确不做以下事情：

- 不删除页面文件
- 不删除前端路由
- 不改后端权限码或菜单权限体系
- 不修改登录页、忘记密码页
- 不把手动输入 URL 的用户重定向到其他页面
- 不统一清理首页内部卡片、手动直达按钮、占位页逻辑，除非它本来就来自侧边栏首屏流程

## 5. 现状入口盘点

### 5.1 侧边栏入口

当前 5 个端的主要显式导航入口都写在各自的 `MainLayout.vue` 中：

- `osg-frontend/packages/student/src/layouts/MainLayout.vue`
- `osg-frontend/packages/mentor/src/layouts/MainLayout.vue`
- `osg-frontend/packages/lead-mentor/src/layouts/MainLayout.vue`
- `osg-frontend/packages/assistant/src/layouts/MainLayout.vue`
- `osg-frontend/packages/admin/src/layouts/MainLayout.vue`

### 5.2 默认首屏入口

当前多个端的根路由仍会把用户带到首页：

- student: `/ -> /dashboard`
- mentor: `/ -> /dashboard`
- lead-mentor: `/ -> /home`
- assistant: `/ -> /home`
- admin: `/ -> /dashboard`

这些重定向如果不改，用户即使看不到侧边栏首页，也仍会在登录后先看到首页。

## 6. 方案选择

### 6.1 方案 A：逐端导航白名单过滤

做法：

- 为每个端定义第一期允许显示的路径白名单
- 侧边栏在渲染前先过滤
- 根路由重定向改为“第一项保留功能”

优点：

- 改动集中
- 风险最小
- 与“用户正常浏览看不到未落地功能”目标完全一致

缺点：

- 需要分别维护 5 端白名单

### 6.2 方案 B：直接手工删菜单项

做法：

- 在每个 `MainLayout.vue` 中硬编码删除不需要的菜单项

优点：

- 最快

缺点：

- 容易漏掉首页按钮和重定向
- 后续二期恢复成本高
- 各端规则分散，难以复查

### 6.3 方案 C：做统一跨端导航配置中心

做法：

- 抽一份多端共享导航真相文件
- 各端从统一配置生成菜单

优点：

- 长期可维护性最高

缺点：

- 本次范围过大
- 超出“只做第一期菜单收口”的最短路径

### 6.4 最终选择

采用方案 A。

## 7. 目标实现模型

### 7.1 单端白名单文件

每个端维护一份简单的第一期导航白名单，例如：

- `student/src/navigation/phase1.ts`
- `mentor/src/navigation/phase1.ts`
- `lead-mentor/src/navigation/phase1.ts`
- `assistant/src/navigation/phase1.ts`
- `admin/src/navigation/phase1.ts`

每份文件只维护两类真相：

1. `PHASE1_VISIBLE_PATHS`
2. `PHASE1_DEFAULT_PATH`

### 7.2 侧边栏过滤

各端 `MainLayout.vue` 保留原始菜单定义，但最终渲染时只输出：

- 在 `PHASE1_VISIBLE_PATHS` 中的菜单
- 与这些菜单强耦合的用户菜单入口

### 7.3 首页隐藏

首页隐藏不是删除页面，而是：

- 侧边栏不再显示首页
- 根路由不再重定向到首页
- 登录后首次进入默认落到一期保留功能

### 7.4 默认落点

建议默认落点如下：

- student: `/positions`
- mentor: `/courses`
- lead-mentor: `/career/positions`
- assistant: `/career/positions`
- admin: `/permission/roles`

## 8. 数据流与行为

本次不改业务数据流，只改导航渲染流：

1. 登录成功
2. 根路由重定向到 `PHASE1_DEFAULT_PATH`
3. `MainLayout` 读取原始菜单定义
4. 使用当前端白名单过滤
5. 只把保留项展示给用户

手动输入 URL 时：

- 仍按当前路由行为进入
- 不做额外拦截
- 不新增 404 / 敬请期待 / 自动跳首页 逻辑

## 9. 错误处理与边界

- 若某端白名单配置了不存在的路径：
  - 开发期由静态测试暴露
  - 不在运行时静默降级
- 若某页面仍有首页卡片或内部 CTA 指向隐藏模块：
  - 只在其来自正常首屏路径时处理
  - 不为了覆盖所有手动直达场景而扩大本次范围

## 10. 测试策略

### 10.1 静态测试

对每个端补最小测试：

- 白名单内菜单仍存在
- 白名单外菜单不再出现在侧边栏源码或渲染结构中
- 根路由 redirect 指向新的默认落点

### 10.2 最小运行时验证

每个端至少验证：

1. 登录成功
2. 首屏落到白名单默认页
3. 侧边栏不再出现隐藏菜单

admin 额外验证：

- 用户菜单仍保留“个人设置”
- “操作日志”仍可见

### 10.3 按端落地验证节奏

真正进入实现时，必须按端逐个推进，不允许一次改多个端后再统一验收。

每完成一个端，都必须执行以下顺序：

1. 完成该端白名单、侧边栏和默认首屏改造
2. 立即跑该端一轮静态测试与最小运行时验证
3. 测完后关闭当前网页进程或浏览器会话，避免旧页面缓存影响结果
4. 再使用 MCP 打开该端真实页面做端到端验证
5. 该端验证通过后，才能进入下一个端

这样可以确保：

- 问题定位只落在当前端
- 不会把多个端的缓存和残留状态混在一起
- 每个端都能形成独立、可信的真实页面验收证据

## 11. 风险

1. 某些页面可能仍存在首页内部快捷入口或遗留跳转按钮，本次默认不全量清理。
2. 部分路径存在别名（如 student 的 `/applications` 与 `/job-tracking`、`/courses` 与 `/myclass`），白名单需要按真实活跃路径和别名一起维护。
3. 若后续二期要恢复隐藏页面，直接调整单端白名单即可，不必回滚路由。
