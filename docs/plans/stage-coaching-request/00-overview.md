## 1. 核心产品口径

产品经理确认后的关键业务事实：

- 学生对一个岗位进入不同面试阶段时，不是原辅导单的阶段状态流转；
- 每次需要阶段辅导时，应重新发起一条独立辅导申请；
- 不同阶段可以分配不同导师；
- 课消详情、最近评分、已上报课消数应归属于具体辅导申请，而不是只归属于岗位申请。

统一建模口径：

```text
岗位申请 application：学生已求职/已投递的岗位父记录
辅导申请 coaching：学生针对某个岗位、某个面试阶段发起的一次辅导子记录
课消记录 class_record：归属于具体 coaching / mock practice / relation / midterm / base course
```

## 3. 需要修正的核心锚点

当前 5/9 主文档中存在一处需要校正的口径：

```text
岗位辅导课消锚点：application
```

产品确认后建议修正为：

```text
岗位辅导课消锚点：coaching request
reference_type = job_coaching
reference_id = coaching_id
```

即：

- `application_id` 用来表示岗位父记录；
- `coaching_id` 用来表示某一次阶段辅导申请；
- 分配导师、待辅导、待分配、课消详情、最近评分、已上报课消数都优先锚定 `coaching_id`。

## 5. 优先级建议

### P0：确认并保护已完成主线

先以现有 Story/Ticket 为主要事实来源，确认当前代码已经落地了哪些内容。

重点：

- 不要把已完成的 S-053~S-056 推倒重来；
- 如果代码里已经做了某些口径校正，以代码和 ticket evidence 为准；
- 后续只补产品口径差异。

### P1：学生端「我的求职」阶段级辅导申请

这是最早提出、且决定后续链路对象边界的入口。

目标：

- 岗位行展示岗位字段；
- 操作区只有「申请辅导」；
- 每次申请辅导生成一条辅导申请；
- 岗位行展开显示多条辅导申请记录；
- 详情、修改、评分、导师都围绕辅导申请。

### P2：求职总览从 application 维度校正到 coaching 维度

班主任/导师/助教求职总览的页面结构可沿用 5/9 主文档。

但行维度、筛选、分配、课消统计应统一锚定具体辅导申请。

### P3：三端课消上报 reference 校正

沿用 S-055/S-056 已抽取的 shared 弹窗与 validator。

重点修正：

```text
岗位辅导 reference_id 从 application_id 改为 coaching_id
```

### P4：模拟应聘管理与其它独立需求

模拟应聘管理、本后台审核、学生资料字段字典化、课程排期弹窗等拆成独立 Story，避免污染岗位辅导主线。

## 6. 后续拆 Story 原则

拆 Story 时按业务对象拆，不按端重复拆：

- `application`：学生已求职岗位；
- `coaching`：一次阶段级岗位辅导申请；
- `mock_practice`：一次模拟应聘/人际关系/期中考试申请；
- `class_record`：一次课消上报；
- `manual_position`：学生自添岗位审核流。

每个 Story 应明确：

- 主数据表；
- 页面入口；
- 读写接口；
- 统计口径；
- 验收标准；
- 与已有 S-053~S-057 的关系。

---

## 7. 关键风险

- **锚点混用风险**：同一功能中同时使用 `application_id` 和 `coaching_id` 统计会导致课消数、最近评分、详情不一致。
- **重复返工风险**：如果继续按 `application` 维度扩展求职总览，后续学生端多阶段辅导上线时会返工。
- **已完成票据保护风险**：S-053~S-056 已有大量实现与 ticket evidence，后续修订应增量补差，不应直接重开全部。
- **接口兼容风险**：已有接口可能以 `applicationId` 命名，后续需要明确是兼容参数还是语义迁移。

## 8. 总结

本轮统一结论：

```text
岗位是一条 application。
阶段辅导申请是多条 coaching。
导师分配、待辅导、课消详情、最近评分、已上报课消数都应围绕 coaching_id。
```

现有 5/9 主文档继续作为主线，但后续需要通过本目录分专题文档进行口径修订与新需求补充。
