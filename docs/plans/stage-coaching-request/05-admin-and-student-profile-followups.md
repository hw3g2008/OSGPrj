# 后台与学生资料后续需求收口

- **范围**：不直接属于阶段级岗位辅导主链或三端课消弹窗的后续需求
- **目的**：避免把独立后台/资料/排期问题混入学生端我的求职与课消主线

---

## 1. 学生端自添岗位审核

### 1.1 用户需求

学生端自添岗位流程：

```text
学生端添加自填岗位后，后台需要审核。
审核有两种处理方式：
1. 如果申请的岗位有重复，就合并岗位；
2. 如果没有这个岗位，就新增岗位。
```

### 1.2 建议业务对象

建议将“学生自添岗位”作为独立审核流处理，不直接写入正式岗位表。

候选对象：

```text
manual_position_submission / osg_manual_position_request
```

第一版如已有表，则优先复用已有表。

### 1.3 状态建议

```text
pending_review  待审核
merged          已合并到已有岗位
created         已新增正式岗位
rejected        已驳回
```

### 1.4 审核方式一：重复合并

后台审核员选择一个已有正式岗位，将学生自添记录合并到该岗位。

结果：

- 自添申请状态改为 `merged`；
- 记录目标 `position_id`；
- 学生的求职申请绑定到已有岗位；
- 不新增重复岗位。

### 1.5 审核方式二：新增岗位

后台审核员确认没有重复岗位后，创建正式岗位。

结果：

- 新增正式岗位记录；
- 自添申请状态改为 `created`；
- 记录新 `position_id`；
- 学生的求职申请绑定到新岗位。

### 1.6 验收标准草案

- 学生提交自添岗位后，后台出现待审核记录。
- 后台可选择“合并已有岗位”。
- 后台可选择“新增岗位”。
- 合并后学生求职记录指向已有岗位。
- 新增后学生求职记录指向新建岗位。
- 审核过程保留操作人和操作时间。

## 2. 后台【导师列表】审核不了变更信息

### 2.1 问题描述

用户反馈：

```text
后台导师列表审核不了变更信息
```

### 2.2 建议先定位

这是独立 bug，不应混入阶段辅导申请主线。

需要确认：

- 变更信息来源表；
- 审核按钮所在页面；
- 当前接口是否存在；
- 是前端按钮不显示、接口报错，还是状态没有流转。

### 2.3 后续处理建议

按 `/fix` 或单独 Story 处理：

- 复现；
- 定位根因；
- 明确审核状态机；
- 修复并补测试。

## 3. 学生端修改信息页面字段走字典

### 3.1 用户需求

```text
学生端修改信息的页面字段都要去字典中的
```

### 3.2 建议范围

学生资料编辑页中，下拉/单选/多选类字段应统一从字典读取。

可能涉及：

- 学校
- 专业
- 年级
- 城市
- 行业偏好
- 岗位分类
- 公司类型
- 求职阶段
- 其它枚举字段

实际字段需以页面代码和原型为准。

### 3.3 验收标准草案

- 学生资料编辑页不再硬编码枚举选项。
- 所有枚举字段从后端字典接口获取。
- 字段展示使用 `dict_label`。
- 提交保存使用 `dict_value`。
- 字典为空时有明确空态或禁用表现。

## 4. 未填写课程排期时强制弹窗

### 4.1 用户需求

```text
导师端 / 班主任端没有填写课程排期的时候，没有强制弹出页面
```

### 4.2 建议业务规则

用户进入导师端/班主任端后，如果当前用户没有填写课程排期，应强制弹出排期页面或弹窗。

需要确认：

- 什么叫“没有填写课程排期”；
- 检查周期是每天、每周还是首次登录；
- 哪些角色强制；
- 是否允许跳过；
- 跳过后是否再次提醒。

### 4.3 建议第一版

第一版可采用：

```text
登录后或进入首页时检查当前用户未来 N 天是否有可用排期。
若没有，弹出课程排期弹窗。
弹窗不可关闭，直到保存成功或产品允许跳过。
```

具体 N 天需产品确认。

## 6. 与主线的边界

这些事项暂不进入阶段级岗位辅导申请主链：

- 自添岗位审核属于岗位库治理；
- 导师变更审核属于后台审核流；
- 学生资料字段字典化属于资料维护体验；
- 课程排期强制弹窗属于排期合规；
- 排期按钮显示属于 UI 修复。

建议分别拆 Story，避免和学生端我的求职 / 求职总览 / 课消上报混在同一批 Ticket 中。
