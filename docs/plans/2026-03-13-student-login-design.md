# Student Login Design

Date: 2026-03-13
Status: Approved for Requirement Landing
Owner: student-module

## Scope

本设计只覆盖 `student` 模块下的登录功能需求落地。

本轮目标：

1. 在不使用 `superpowers` 的前提下，按 `.claude` 下的 RPIV / brainstorm 真源链路明确学生端登录功能需求。
2. 把“学生端登录功能”收口为一个完整功能闭环，而不是把整个学生端业务页面一起展开。
3. 只停在需求层，不继续进入 `split story / split ticket / implement`。

## Truth Source

本设计采用以下真源与参考输入：

1. `.claude/project/config.yaml`
   - `prd_process.truth_source`
   - `prd_process.module_prototype_map.student`
2. `osg-spec-docs/source/prototype/index.html`
3. `osg-spec-docs/source/prototype/PRD_Student.html`
4. `osg-spec-docs/docs/02-requirements/functional/REQ-FUN-001_student.md`
5. `osg-spec-docs/docs/09-testing/test-cases/TST-STU-001_login.md`
6. `osg-spec-docs/docs/06-api/openapi.yaml`

现有前端实现只作为现状参考，不作为需求真源。

## In Scope

本轮 `student` 登录功能必须覆盖以下完整能力：

1. 登录页完整入口界面
2. 用户名或邮箱登录
3. 密码输入与显示/隐藏
4. 记住我
5. 忘记密码完整流程
6. 登录成功后的登录态保存与首页跳转
7. 登录失败后的错误反馈
8. 冻结态、已结束态入口与受限界面
9. 表单校验、loading、防重复提交、Enter 提交
10. 登录页桌面端与移动端可用性

## Out Of Scope

本轮不覆盖以下内容：

1. 登录后首页、岗位、课程、资源、个人中心等业务功能细节
2. 注册功能
3. 学生端除登录页外的其他业务模块需求落地
4. Story / Ticket / 实现 / 验证阶段工作

## Functional Requirements

### FR-LOGIN-001 页面入口

学生端必须提供完整登录入口页面，至少包含：

1. 品牌标识
2. 平台介绍文案
3. 特性列表
4. 用户名/邮箱输入框
5. 密码输入框
6. 登录按钮
7. 记住我
8. 忘记密码入口
9. 冻结态、已结束态演示入口

### FR-LOGIN-002 表单输入与校验

登录表单必须满足：

1. 用户名字段支持用户名或邮箱输入
2. 密码字段支持显示/隐藏切换
3. 用户名为空时提示“请输入用户名或邮箱”
4. 密码为空时提示“请输入密码”
5. 邮箱格式非法时提示有效邮箱格式错误
6. 密码长度不足时提示长度错误
7. 表单提交前必须先经过前端校验

### FR-LOGIN-003 登录提交

登录请求必须形成完整提交闭环：

1. 提交 `username`
2. 提交 `password`
3. 提交学生端身份标识 `client_type=student`
4. 勾选记住我时提交记住登录状态字段
5. 提交期间页面进入 loading 状态
6. 提交期间禁止重复点击

### FR-LOGIN-004 登录成功处理

登录成功后系统必须：

1. 保存 token
2. 保存过期时间
3. 保存当前用户信息
4. 建立可恢复的登录态
5. 跳转到学生端首页
6. 如果存在 `redirect` 参数，则优先跳转目标地址

### FR-LOGIN-005 登录失败处理

登录失败后系统必须留在登录页，并按错误场景给出反馈：

1. 用户名或密码错误
2. 账号禁用
3. 账号冻结
4. 账号锁定
5. 网络错误

同时必须满足：

1. 不泄露“用户不存在”这类敏感信息
2. 密码错误后允许用户快速重试
3. 不产生重复可见错误提示

### FR-LOGIN-006 记住我

记住我必须构成真实功能差异：

1. 未勾选时使用默认登录有效期
2. 勾选后使用更长登录有效期
3. 重新打开页面后可恢复登录状态

### FR-LOGIN-007 忘记密码

忘记密码必须是完整闭环，而不是占位入口：

1. 输入注册邮箱
2. 发送验证码
3. 验证码倒计时与重发
4. 输入验证码
5. 设置新密码
6. 确认新密码
7. 提交重置
8. 重置成功后返回登录页

该流程必须具备：

1. 邮箱格式校验
2. 验证码长度校验
3. 两次密码一致性校验
4. 验证码过期/错误处理
5. 不泄露邮箱是否存在

### FR-LOGIN-008 冻结态与已结束态

冻结态与已结束态属于登录功能范围，系统必须支持：

1. 通过登录页入口进入受限界面
2. 明确展示当前受限状态说明
3. 限制用户可访问能力
4. 保留受限模式下允许查看的内容范围

### FR-LOGIN-009 交互体验

登录页必须具备以下交互体验：

1. Enter 键提交表单
2. 提交中的 loading 可见
3. 错误提示清晰可见
4. 响应式布局可用
5. 页面刷新后状态恢复正确
6. 不出现死链、无响应按钮、空白态跳转错误

### FR-LOGIN-010 接口能力闭环

登录功能在需求层必须覆盖以下认证接口能力：

1. 登录
2. 获取当前登录用户
3. 发送验证码
4. 校验验证码
5. 重置密码
6. 登出

## Shared Reuse Requirements

学生端登录功能必须优先复用 `@osg/shared`，不得在 `student` 包内单独复制一套认证基础能力。

### AR-SHARED-001 页面与底座分层

`student` 登录页只负责：

1. 页面展示
2. 表单交互
3. 状态切换
4. 跳转编排

认证底座能力不在 `student` 页面内重复实现。

### AR-SHARED-002 共享认证能力

以下能力必须统一由 `@osg/shared` 承担：

1. 登录
2. 登出
3. 获取当前用户
4. 发送验证码
5. 校验验证码
6. 重置密码
7. token 读写
8. 用户信息缓存
9. 登录态清理
10. 过期处理
11. 统一错误提示策略

### AR-SHARED-003 学生端身份参数

学生端调用共享认证能力时，必须显式带上 `client_type=student`，不能由页面层隐式猜测。

### AR-SHARED-004 共享状态策略

记住我、登录有效期、用户缓存、清空登录态等能力，必须以共享能力形式复用，避免 mentor / assistant / lead-mentor / student 四端重复实现。

### AR-SHARED-005 Shared 缺口归属

如果现有 `@osg/shared` 的 auth 能力、字段定义或导出口径无法支撑本轮完整登录功能，这些缺口应归为“补齐 shared 能力”，而不是在 `student` 端做临时绕过。

## Components

本轮需求涉及的主要组件/能力边界如下：

1. Student Login Page
2. Forgot Password Modal / Step Flow
3. Restricted State View
4. Shared Auth API
5. Shared Auth Storage
6. Shared Request / Error Handling

## Data Flow

### 登录主链

1. 用户进入登录页
2. 输入用户名/邮箱和密码
3. 前端校验通过
4. 调用共享登录能力
5. 共享层提交认证请求
6. 登录成功后共享层返回 token 与用户信息
7. 页面层保存登录态并跳转首页

### 忘记密码链

1. 用户打开忘记密码入口
2. 输入邮箱
3. 发送验证码
4. 输入验证码
5. 设置新密码并确认
6. 调用共享重置密码能力
7. 重置成功后返回登录页

### 受限态链

1. 用户通过受限态入口进入受限模式
2. 系统展示冻结或已结束状态说明
3. 系统限制能力范围

## Error Handling

本轮需求要求明确以下错误处理：

1. 字段为空
2. 邮箱格式错误
3. 密码长度不足
4. 用户名或密码错误
5. 账号禁用
6. 账号冻结
7. 账号锁定
8. 验证码错误
9. 验证码过期
10. 两次密码不一致
11. 网络错误
12. token 失效

同时必须遵守：

1. 单个错误路径只能有一个用户可见错误 owner
2. 不允许共享层与页面层重复弹错

## Acceptance

当以下条件同时满足时，可以认为“学生端登录功能需求”已落地：

1. 登录页完整能力已全部纳入需求
2. `shared` 复用边界已明确
3. in-scope / out-of-scope 已明确
4. 需求不再依赖当前最简实现作为口径
5. 后续实现团队可以仅凭本需求继续进入下游流程

## Notes

1. 本设计只负责需求落地，不推进实现。
2. 本设计不使用 `superpowers`。
3. 本设计默认后续流程另行启动，不在本轮展开。
