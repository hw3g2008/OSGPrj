# 学生端正式数据流收口实施计划

> Goal: 按 `docs/plans/five-end-course-flow/2026-03-27-student-formal-dataflow-closure-design.md` 的设计，收口学生端首批保留页面的正式数据来源、写入链路、错误处理与跨端一致性。
>
> Scope: `岗位信息 / 我的求职 / 模拟应聘 / 我的课程 / 基本信息`
>
> Strategy: 先去掉正式写链中的假成功与静默降级，再收口两类辅导申请的页面语义与主链归属，最后完成统一课程记录回看与跨端闭环验证。

## 0. 执行原则

1. 不恢复已隐藏的非首批页面。
2. 正式链成功才允许返回成功；正式链失败必须明确失败。
3. 不允许通过 shadow fallback 维持“看起来成功”。
4. 可见页面的数据来源必须可解释。
5. 学生端与班主任/后台对同一业务对象必须共享同一条正式主链。

## 1. Workstream A: 收口正式链前置条件

目标：所有会写正式业务链的学生端操作，在写入前都先验证正式 `student_id` 与业务前置条件。

任务：

1. 盘点学生端 5 个保留页中所有正式写动作：
   - 岗位投递
   - 岗位进度更新
   - 真实岗位辅导申请
   - 模拟应聘辅导申请
   - 课程记录评价
   - 资料更新
2. 梳理这些动作当前依赖的正式身份解析逻辑与 fallback 分支。
3. 在后端服务层建立统一规则：
   - 能解析正式身份时走正式主链
   - 不能解析时返回明确业务失败
4. 前端为这些失败建立统一受控提示或受限态，不显示技术性异常。

验收：

- 不再因为身份缺失而静默写入旧学生独立表
- 不再因为主链未打通而提示成功

## 2. Workstream B: 岗位信息与我的求职主链收口

目标：`岗位信息` 与 `我的求职` 共享同一套正式求职主状态。

任务：

1. 审核 `PositionServiceImpl` 中以下分支：
   - 正式读取
   - 影子表读取
   - apply/progress/coaching 写入
   - 主链异常吞掉的兼容分支
2. 去掉以下行为：
   - 主链失败后继续返回成功
   - 学生页读影子表掩盖正式链失败
3. 保留 `公共岗位池` 与 `学生自添待审核岗位` 的展示，但显式区分来源与可操作性。
4. 校验班主任/后台读取的求职主状态与学生端读取口径一致。
5. 清理岗位页中硬编码的默认业务值，例如默认投递日期。

验收：

- 学生端岗位投递、进度更新、真实岗位辅导申请只写正式求职主链
- 我的求职页只显示正式求职主状态
- 不再出现“学生端成功、后台无数据”的假健康

## 3. Workstream C: 模拟应聘页面语义收口

目标：`模拟应聘` 页面只保留明确的 mock 业务入口，不再保留语义模糊的混合申请入口。

任务：

1. 审核当前页面中的申请入口：
   - `practice-request`
   - `class-request`
2. 把现有可见入口映射到已确认的两类业务语义：
   - 真实岗位辅导申请
   - 模拟应聘辅导申请
3. 对语义不清的入口做以下三选一收口：
   - 改名为明确业务意图
   - 合并进已有 mock 申请入口
   - 从当前首批可见范围中移除
4. 保证 mock 页只承担模拟应聘主链的申请与状态展示。
5. 校验学生发起 mock 申请后，班主任/后台能从正式 mock 主链读取到同一申请。

验收：

- 模拟应聘页不再存在“既像课程申请又像模拟应聘申请”的混合入口
- 学生端 mock 申请状态与班主任/后台读取同源

## 4. Workstream D: 统一课程记录回看

目标：`我的课程` 成为唯一课程记录回看页，统一回看两类辅导申请后的授课结果。

任务：

1. 审核课程记录页当前读取主表与 fallback 逻辑。
2. 明确课程记录回看规则：
   - 只显示后台审核通过的正式课程记录
   - 每条记录带来源标签
     - 真实岗位辅导
     - 模拟应聘辅导
3. 校验导师多次授课、多次提交记录的场景在学生端可完整回看。
4. 校验后台审核后学生端课程记录页能稳定同步。
5. 处理正式记录为空时的业务空态，不用影子数据或假文案填充。

验收：

- 学生端课程记录统一回看成立
- 两类辅导申请都能汇入课程记录页
- 未审核记录不提前暴露给学生

## 5. Workstream E: 基本信息正式写链收口

目标：资料页保留统一视图，但写入只允许正式可解释的数据流。

任务：

1. 审核资料页当前的即时字段、审核字段与 fallback pending 表逻辑。
2. 明确字段分组：
   - 即时生效字段
   - 审核后生效字段
3. 即时字段只在正式允许范围内直接更新。
4. 审核字段统一走正式变更申请链。
5. 正式身份未打通时返回明确失败，不再回退旧 pending 表并提示成功。

验收：

- 资料页每个可编辑字段都有明确生效路径
- 学生端看到的待审核状态与后台审核链一致

## 6. 测试计划

### 6.1 后端服务测试

补充或调整以下服务测试：

- `PositionServiceImplTest`
  - 主链成功时返回成功
  - 主链失败时返回明确失败
  - 不再吞掉正式链写入异常
- `StudentMockPracticeServiceImplTest`
  - mock 申请入口与正式 mock 主链一致
  - 混合入口收口后的行为稳定
- `StudentCourseRecordServiceImplTest`
  - 只回看审核通过记录
  - 来源标签正确
- `StudentProfileServiceImpl` 相关测试
  - 正式资料链失败时不再假成功

### 6.2 前端页面测试

补充或调整学生端测试，覆盖：

- 正常态
- 空态
- 受限态
- 正式写链失败提示
- 不出现硬编码假业务值
- 不出现技术性报错

重点页面：

- `positions`
- `applications`
- `mock-practice`
- `courses`
- `profile`

### 6.3 跨端链路验证

至少验证两条闭环：

1. 真实岗位辅导申请闭环
2. 模拟应聘辅导申请闭环

每条闭环都要验证：

- 学生端发起
- 班主任/助教可读
- 导师可提交多次课程记录
- 后台审核
- 学生统一回看

## 7. 建议实施顺序

1. 先改后端正式链前置条件与错误处理
2. 再改岗位信息/我的求职正式主链
3. 再改模拟应聘页入口语义
4. 再改课程记录统一回看
5. 最后改资料页正式写链
6. 收尾补齐前后端测试与跨端验证

## 8. 完成定义

满足以下条件才算本轮完成：

1. 学生端 5 个保留页无技术性报错
2. 无假成功
3. 无静默 shadow 降级
4. 双申请链可闭环
5. 课程记录统一回看成立
6. 学生端与班主任/后台口径一致
