# 学生端岗位真源收口设计补充

日期：2026-03-27  
状态：设计已确认，待进入实施计划  
关联主设计：`docs/plans/five-end-course-flow/2026-03-27-student-formal-dataflow-closure-design.md`

## 1. 背景

在上一轮“学生端正式数据流收口”设计中，已经确认：

- 学生端保留页必须做到“对的就是对的，错的就是错的”
- 正式链失败不能假成功
- `我的求职` 只读正式 `osg_job_application`
- `模拟应聘` 页不再保留语义模糊的混合 `class-request` 入口

当前剩下的关键问题集中在学生端岗位页：

- `我的求职` 已切到正式主链
- 但岗位列表本身仍主要来自旧学生岗位池 `osg_student_job_position`
- 同一个岗位在“岗位页”和“我的求职”里的状态来源还不完全一致

这会导致学生端虽然表面可用，但仍无法清楚回答“当前岗位列表是不是来自正式渠道”。

## 2. 设计目标

本补充设计只处理岗位相关收口，目标如下：

1. 学生端岗位列表只保留正式可解释来源
2. `我的求职` 与岗位页上的正式状态口径一致
3. 学生自添待审核岗位不再伪装成正式可操作岗位
4. 页面失败时保持业务失败提示，但不出现技术性炸错

## 3. 非目标

本轮不做以下事情：

- 不把班主任端和后台端岗位视图一起重构
- 不一次性重做全部岗位库建模
- 不恢复旧学生岗位池作为学生端主展示来源
- 不处理资料页或课程页的进一步真源收口

## 4. 推荐方案

采用“方案 2：岗位列表只保留正式公共岗位 + 学生自添待审核岗位”。

核心思路：

- 学生端岗位页不再把旧学生岗位池 `osg_student_job_position` 当作主列表真源
- 岗位列表只保留两类可解释来源
  - 正式公共岗位：`osg_position`
  - 学生自添待审核岗位：`osg_student_position`
- `我的求职` 继续只读正式 `osg_job_application`
- 公共岗位上的投递/进度/辅导状态，尽量与正式 `osg_job_application` 对齐

## 5. 岗位列表真源设计

### 5.1 列表来源

学生端岗位列表只允许出现两类条目：

- 正式公共岗位
  - 来源：`osg_position`
  - 条件：处于正式可见状态
- 学生自添待审核岗位
  - 来源：`osg_student_position`
  - 条件：属于当前学生本人，且尚未审核通过

明确去掉的来源：

- `osg_student_job_position` 不再作为学生端岗位页主清单来源

旧学生岗位池如果仍保留，也只能用于迁移校验或内部兼容判断，不允许再决定学生端可见岗位列表。

### 5.2 页面展示语义

岗位列表中每条岗位都必须能解释清楚来源：

- 正式公共岗位：可进入正式求职流程
- 学生自添待审核岗位：仅表示“已提交审核，尚未进入正式岗位库”

审核通过后，自添岗位应进入正式公共岗位口径，不继续停留在学生私有展示层。

## 6. 状态与主链对齐设计

### 6.1 我的求职

`我的求职` 保持不变，继续只读正式 `osg_job_application`。

它是学生求职状态的唯一主视图。

### 6.2 岗位页公共岗位状态

对正式公共岗位，岗位页上的以下状态应尽量由正式主链反映：

- 是否已投递
- 当前阶段
- 辅导状态

即：

- 正式主链无申请记录：显示 `未投递`
- 正式主链有申请记录：显示正式状态
  - `已投递`
  - `某阶段中`
  - `辅导待审批`
  - `辅导中`

不再允许岗位页继续主要依赖旧学生岗位池里的 `applied / progressStage / coachingStatus` 作为公共岗位状态真源。

### 6.3 待审核岗位状态

学生自添待审核岗位：

- 不进入求职主状态
- 不显示正式投递/进度/辅导状态
- 仅显示 `待审核`

这样可以避免“待审核岗位看起来已经进入正式求职链”的误导。

## 7. 页面交互与受限态

### 7.1 正式公共岗位

允许操作：

- 投递
- 取消投递
- 更新进度
- 申请真实岗位辅导
- 收藏

但前提是：

- 当前账号已进入正式业务链
- 当前岗位是正式可见公共岗位

### 7.2 学生自添待审核岗位

允许操作：

- 查看
- 收藏（如保留轻量整理能力）

不允许操作：

- 投递
- 更新进度
- 申请辅导

页面应明确提示，例如：

- 该岗位正在审核，通过后将进入正式岗位库

### 7.3 错误处理

本轮岗位页错误处理遵循以下原则：

- 正式写链失败时不假成功
- 不先乐观更新再静默回滚
- 不显示技术性异常栈
- 只显示明确业务失败提示
- 页面不出现未捕获 Promise 或控制台炸错

## 8. 与跨端口径的关系

本轮不直接重构班主任端与后台岗位视图，但要保持可衔接性：

- 学生端 `我的求职` 使用的正式主状态，仍与班主任端/后台使用的 `osg_job_application` 一致
- 本轮先收学生端岗位列表真源
- 下一阶段再继续做跨端岗位视图来源统一

## 9. 测试与验收

### 9.1 通过标准

满足以下条件才算本补充设计落地完成：

1. 学生端岗位列表只展示正式公共岗位与学生自添待审核岗位
2. 岗位页公共岗位状态与 `我的求职` 正式主状态一致
3. 待审核岗位不再出现正式求职动作
4. 正式链失败时页面不假成功、不炸错

### 9.2 最小验收路径

至少验证两条路径：

1. 正式公共岗位
   - 学生投递
   - 岗位页状态变化
   - `我的求职` 页显示同一正式状态
2. 学生自添待审核岗位
   - 学生提交岗位
   - 岗位页显示 `待审核`
   - 不可投递、不可更新进度、不可申请辅导

## 10. 结论

本补充设计的本质不是“再修一次岗位页样式”，而是把岗位页本身从“旧学生岗位池展示面”收成“正式公共岗位 + 正式待审核岗位”的可解释真源。

这样做完后，学生端就能明确回答：

- 岗位列表来自哪里
- 哪些岗位能进入正式求职链
- 哪些岗位只是审核中的提交记录
- 岗位页状态为什么和 `我的求职` 一致
