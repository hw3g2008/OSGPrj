## 用户故事一：管理员小陈的日常运营

**背景：**  
管理员小陈负责 OSG 系统的日常运营，包括账号与权限维护、学员与导师建档、合同管理、岗位发布、模拟应聘管理、课程审核和后台运营配置。她是整个平台数据初始化和业务流转的起点。

### 第一幕：开工登录与系统检查

新的一天开始，小陈先登录 Admin 后台，确认系统可以正常进入。登录后，她会先查看权限配置、后台用户管理和基础数据管理，确保角色权限、字典数据和后台账号都处于可用状态，为当天运营工作做好准备。

- 流程：后台登录页 → 登录 / 忘记密码 → 权限管理 → 权限配置 → 后台用户管理 → 基础数据管理

### 第二幕：新班主任与导师入职

今天有新的老师加入团队，小陈需要先在系统里完成导师档案维护。她进入“导师列表”新增导师，填写老师的姓名、邮箱、类型和专长信息。完成后，她还要检查导师排期管理，确认后续能够为该老师安排课程与服务任务。

- 流程：后台 → 用户中心 → 导师列表 → 新增
- 流程：后台 → 用户中心 → 导师排期管理

### 第三幕：新学员入学建档

一位新学员报名成功后，小陈需要在系统中完成入学建档。她在“学员列表”中创建学员账号，录入姓名、邮箱、手机、学校、专业和求职方向，同时为学员分配班主任李老师和助教小赵。随后，小陈还会录入初始培训合同，确保学员在后续教学和求职服务中拥有完整档案。

- 流程：后台 → 用户中心 → 学员列表 → 新增
- 流程：后台 → 用户中心 → 学员列表 → 查看详情
- 流程：后台 → 用户中心 → 合同管理 → 新增合同

### 第四幕：确认跨端同步

为了确保建档成功，小陈会进一步确认新学员信息是否已经同步到其他角色端。学员应当能够用默认密码登录学生端，班主任李老师和助教小赵也应在各自的学员列表中看到这名新学员。这样，小陈才能确认后台录入的数据已经真正进入业务链路。

- 流程：学生端 → 登录
- 流程：班主任端 → 学员列表
- 流程：助教端 → 学员列表

### 第五幕：合同续签与数据导出

除了新学员入学，小陈还要处理老学员合同续签。她在“合同管理”中查看学员合同详情，完成续签操作，并在必要时导出合同数据给财务或运营团队使用。她需要确保金额、课时、币种、起止时间和状态都准确无误。

- 流程：后台 → 用户中心 → 合同管理 → 合同详情
- 流程：后台 → 用户中心 → 合同管理 → 续签
- 流程：后台 → 用户中心 → 合同管理 → 导出

### 第六幕：发布新岗位

合作企业发来了新的招聘需求，小陈需要第一时间把岗位发布到系统里。她进入“岗位信息”新增岗位，填写公司、岗位名称和地点等信息。发布完成后，学生端、班主任端和助教端都应能看到这个岗位，确保求职服务链路开始运转。

- 流程：后台 → 求职中心 → 岗位信息 → 新增
- 流程：学生端 → 岗位信息
- 流程：班主任端 → 岗位信息
- 流程：助教端 → 岗位信息

### 第七幕：查看学员求职与模拟应聘情况

为了掌握平台整体运营状态，小陈会进入“学生自添岗位”“学员求职总览”和“模拟应聘管理”，查看学员们最近新增了哪些自投岗位、整体投递进度如何、是否有新的模拟应聘申请等待分配。她需要基于这些信息做出后续的管理动作。

- 流程：后台 → 求职中心 → 学生自添岗位
- 流程：后台 → 求职中心 → 学员求职总览
- 流程：后台 → 求职中心 → 模拟应聘管理

### 第八幕：课程审核与个人运营

一天结束前，小陈还会到“课程记录”中查看全平台教学记录，对导师提交的课程记录进行审核，并在“操作日志”中确认重要后台动作已留痕。必要时，她也会在“个人设置”中维护自己的后台资料。

- 流程：后台 → 教学中心 → 课程记录
- 流程：后台 → 个人中心 → 操作日志
- 流程：后台 → 个人中心 → 个人设置

## 用户故事二：学员小王的求职之旅

**背景：**  
学员小王是平台的核心服务对象。他会在系统里浏览岗位、管理自己的求职进度、申请模拟应聘、查看课程记录并维护个人资料。平台中导师、班主任、助教和管理员围绕他的求职与学习行为开展协作。

### 第一幕：登录进入平台

小王第一次使用学生端时，需要输入账号密码登录系统。如果忘记密码，他也需要能顺利进入找回密码流程。只有登录顺畅，他后续的求职与学习动作才能展开。

- 流程：学生端登录页 → 登录 / 忘记密码

### 第二幕：浏览和收藏岗位

登录后，小王首先会进入“岗位信息”查看平台最新发布的岗位。他能看到管理员新发布的岗位，也能把感兴趣的岗位先收藏起来，方便后续集中处理和比较。

- 流程：学生端 → 求职中心 → 岗位信息 → 浏览
- 流程：学生端 → 求职中心 → 岗位信息 → 收藏

### 第三幕：开始投递并记录进度

找到合适的岗位后，小王会把岗位标记为“已投递”，同时填写投递日期和投递方式。随着求职推进，他还会继续更新每个岗位的面试进度，例如 Online Test、First Round 等，让平台能沉淀完整的求职轨迹。

- 流程：学生端 → 求职中心 → 岗位信息 → 标记已投递
- 流程：学生端 → 求职中心 → 岗位详情 / 求职记录 → 更新面试进度

### 第四幕：申请岗位辅导与自添岗位

面对重点岗位时，小王希望得到老师帮助，因此会提交岗位辅导申请。与此同时，如果他在外部渠道发现了平台岗位库中没有的新机会，也可以自行补录岗位，让后台和老师端都能看到他的真实求职动作。

- 流程：学生端 → 求职中心 → 岗位信息 / 我的求职 → 申请辅导
- 流程：学生端 → 求职中心 → 岗位信息 / 我的求职 → 手动添加岗位
- 流程：后台 → 求职中心 → 学生自添岗位

### 第五幕：查看“我的求职”

为了清楚掌握自己当前的求职状态，小王会进入“我的求职”页面，按“已收藏”“已投递”“面试中”等分类查看自己的岗位。必要时，他会按状态进行筛选，快速找到某一个阶段的机会。

- 流程：学生端 → 求职中心 → 我的求职
- 流程：学生端 → 求职中心 → 我的求职 → 按状态筛选

### 第六幕：申请模拟应聘

在准备正式面试之前，小王会到“模拟应聘”里提交模拟面试申请，填写类型和申请原因。提交后，他希望看到状态从“待分配”逐步变化为“已确认”，并明确知道是哪位导师会负责这次辅导。

- 流程：学生端 → 求职中心 → 模拟应聘 → 提交申请
- 流程：后台 → 求职中心 → 模拟应聘管理 → 分配导师
- 流程：导师端 → 模拟应聘管理 → 确认
- 流程：学生端 → 求职中心 → 模拟应聘 → 查看状态

### 第七幕：查看课程记录并反馈

小王除了求职，也会持续参加课程。他会在“课程记录”中查看自己已经上过的课程，包括日期、导师、课程类型和时长。课程结束后，他还可以对课程进行评分和反馈，帮助老师了解自己的学习体验。

- 流程：学生端 → 学习中心 → 课程记录
- 流程：学生端 → 学习中心 → 课程记录 → 评价 / 反馈
- 流程：导师端 → 课程记录 → 查看学员评价

### 第八幕：维护基本信息

随着学习过程推进，小王可能会更新自己的手机号等基本资料。他希望在学生端可以直接修改可编辑字段，并且这些修改能同步给班主任和老师查看，避免信息不一致。

- 流程：学生端 → 个人中心 / 基本信息 → 查看
- 流程：学生端 → 个人中心 / 基本信息 → 编辑并保存
- 流程：班主任端 → 学员详情

## 用户故事三：导师张老师的一天

**背景：**  
导师张老师主要负责学员的课程辅导与模拟应聘支持。他需要在导师端处理课程记录、查看学员求职进度、确认模拟面试任务，并管理自己的排期和个人资料。

### 第一幕：登录开始工作

张老师每天开始工作前，需要登录导师端进入系统。如果忘记密码，也需要能从登录页顺利进入找回流程，保证工作不中断。

- 流程：导师端登录页 → 登录 / 忘记密码

### 第二幕：查看并创建课程记录

完成一节课程后，张老师会进入“课程记录”查看自己的历史授课记录，并为学员新建一条课程记录。他需要填写学员姓名、课程内容、时长和课程类型，让平台完整记录教学过程。

- 流程：导师端 → 教学中心 → 课程记录 → 查看列表
- 流程：导师端 → 教学中心 → 课程记录 → 新建

### 第三幕：等待后台审核并确认多端可见

课程记录提交后，张老师知道这条记录会先进入“待审核”状态，等待管理员审核通过。审核完成后，学员端、班主任端和助教端都应看到同一条课程记录，确保教学信息在各角色之间保持一致。

- 流程：导师端 → 教学中心 → 课程记录 → 提交
- 流程：后台 → 教学中心 → 课程记录 → 审核
- 流程：学生端 → 课程记录
- 流程：班主任端 → 课程记录
- 流程：助教端 → 课程记录

### 第四幕：跟进学员求职进度

除了授课，张老师也会进入“学员求职总览”查看自己负责学员当前的投递和面试进展。这样他能更有针对性地安排辅导内容，而不是只关注课堂本身。

- 流程：导师端 → 求职中心 → 学员求职总览

### 第五幕：处理模拟应聘任务

如果管理员给张老师分配了一条模拟应聘申请，他会在“模拟应聘管理”中查看该任务，并进行确认。确认之后，学员端也应看到状态已更新，知道后续由张老师来负责。

- 流程：导师端 → 求职中心 → 模拟应聘管理 → 查看任务
- 流程：导师端 → 求职中心 → 模拟应聘管理 → 确认
- 流程：学生端 → 求职中心 → 模拟应聘 → 查看更新状态

### 第六幕：查看个人信息与课程排期

为了安排自己的教学工作，张老师还会查看“基本信息”和“课程排期”。他希望导师端展示的排课内容与管理员在后台看到的数据一致，避免排期冲突或信息偏差。

- 流程：导师端 → 个人中心 → 基本信息
- 流程：导师端 → 教学中心 / 个人中心 → 课程排期
- 流程：后台 → 用户中心 → 导师排期管理

## 用户故事四：班主任李老师管理学员

**背景：**  
班主任李老师负责学员日常跟进，重点关注学员档案、合同、岗位投递进展、模拟应聘安排和课程记录。她的工作重点是横向掌握学员全貌，并与导师、助教、管理员协同。助教小赵的核心模块与班主任端一致，因此这条故事线也可以作为助教端的镜像路径。

### 第一幕：登录进入班主任工作台

李老师每天开始工作时，需要先登录班主任端，必要时也要能走通忘记密码流程。与此同时，助教小赵也会进入助教端，准备协同查看学员情况。

- 流程：班主任端登录页 → 登录 / 忘记密码
- 流程：助教端登录页 → 登录 / 忘记密码

### 第二幕：查看学员列表与学员详情

进入系统后，李老师首先会打开“学员列表”，查看自己所负责的学员。她点击进入小王的详情页后，希望看到完整的个人信息、合同情况和课时数据，从而快速掌握这名学员的整体状态。助教端也应能看到同样的所辖学员范围。

- 流程：班主任端 → 学员中心 → 学员列表
- 流程：班主任端 → 学员中心 → 学员详情
- 流程：助教端 → 学员中心 → 学员列表

### 第三幕：关注岗位和求职进度

李老师会进入“岗位信息”和“学员求职总览”，查看当前平台上的岗位池，以及自己所辖学员的投递、收藏和面试进度。她希望学员一旦新增投递记录，自己刷新后就能立刻看到最新变化，以便及时跟进。

- 流程：班主任端 → 求职中心 → 岗位信息
- 流程：班主任端 → 求职中心 → 学员求职总览
- 流程：学生端 → 求职中心 → 标记投递新岗位
- 流程：班主任端 → 求职中心 → 学员求职总览 → 刷新查看
- 流程：助教端 → 求职中心 → 岗位信息 / 学员求职总览

### 第四幕：查看模拟应聘安排

当学员提交了模拟应聘申请后，李老师也需要从“模拟应聘管理”中了解当前状态，确认学员是否已被分配导师、是否已确认安排。助教小赵也会在自己的端上查看相同的数据，用于协助跟进。

- 流程：班主任端 → 求职中心 → 模拟应聘管理
- 流程：助教端 → 求职中心 → 模拟应聘管理

### 第五幕：查看课程记录并核对多端一致性

除了求职进展，李老师还会关注学员的学习进度。她进入“课程记录”查看所辖学员的课程列表，并希望看到的内容与导师端、助教端完全一致。这样她才能基于同一份教学记录做跟进和沟通。

- 流程：班主任端 → 教学中心 → 课程记录
- 流程：助教端 → 教学中心 → 课程记录
- 流程：导师端 → 教学中心 → 课程记录

### 第六幕：查看自己的资料和排期

在日常学员管理之外，李老师还需要查看自己的基本信息和课程排期，确保个人工作安排清晰可控，方便后续与导师和助教协同。

- 流程：班主任端 → 个人中心 → 基本信息
- 流程：班主任端 → 教学中心 / 个人中心 → 课程排期
