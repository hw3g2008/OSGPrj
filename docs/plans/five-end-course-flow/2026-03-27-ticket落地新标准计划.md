# Ticket 落地新标准计划

## 1. 文档定位

本文档是当前这一轮 `五端课程申请 / 课程记录流转` 的试行执行标准。

它的目标只有两个：

- 让当前轮次的 ticket 拆分与执行能够直接承接测试先行
- 让 ticket 完成后尽可能直接进入验收，而不是再靠人工补理解

本文档不是全项目框架规范，也不是对既有 RPIV 工作流的直接替换。它只作为本轮交付必须遵守的人工执行标准，为后续框架吸收提供试运行样本。

## 2. 适用范围与边界

### 2.1 适用范围

本文档只适用于以下 workstream：

- `Workstream C1`：学生端 `我的求职`
- `Workstream C2`：学生端 `模拟应聘`
- `Workstream C3`：学生端 `课程记录`
- `Workstream D`：导师提交与后台审核链
- `Workstream E`：助教端去误导
- `Workstream F`：跨端验证与验收

### 2.2 不适用范围

本文档不做以下事情：

- 不 retroactively 重写历史已完成 ticket
- 不直接修改现有 `split-ticket / deliver-ticket / verify` 技能规则
- 不将本轮试行标准直接提升为全项目默认框架

### 2.3 与现有实现计划的关系

[2026-03-27-five-end-course-application-flow-alignment-implementation-plan.md](/Users/hw/workspace/OSGPrj/docs/plans/five-end-course-flow/2026-03-27-five-end-course-application-flow-alignment-implementation-plan.md) 继续定义：

- 做什么
- 按什么业务流转做
- 各阶段先后顺序

本文档负责定义：

- ticket 怎么拆才可执行
- test 怎么先行
- 什么条件下算 done
- 什么条件下允许进入验收

## 3. 核心原则

### 3.1 行为单元原则

本轮执行中，`一个 ticket = 一个可验收行为单元`。

ticket 既不是行级代码片段，也不是一整个模糊功能块。一个 ticket 完成后，必须能用一句完整业务话语描述其已交付行为，例如：

- 学生可以从岗位信息页发起真实岗位辅导申请
- 我的求职页只展示申请追踪与摘要，不展示课程记录详情
- 模拟应聘页不会重新暴露 `class-request` 主流程
- 助教通过 `job-overview` carrier route 只能看到 assistant ownership 范围

### 3.2 测试先行原则

本轮不允许“先实现，再补一个回归测试”的做法。

每个行为单元都必须先定义验收级 test case，再进入实现。实现 ticket 的职责不是“自由发挥写代码”，而是“把已定义的验收用例打绿”。

### 3.3 验收前置原则

本轮的目标不是“代码写完后再想怎么验收”，而是让测试资产本身成为验收资产。

任何 ticket 若没有足够细致的验收级 test case，不允许进入实现阶段。

## 4. Ticket 粒度标准

### 4.1 单 ticket 覆盖边界

每个 ticket 最多覆盖 `1-2` 个强相关 AC，不允许将多个链路阶段混进同一个 ticket。

以下阶段不得在同一 ticket 中混合交付：

- 学生发起申请
- 班主任 / 后台分配导师
- 导师提交课程记录
- 后台审核
- 学生回看与评价

### 4.2 Ticket 类型顺序

每个 workstream 必须按以下顺序组织：

1. `test-design ticket`
2. `backend / frontend / frontend-ui` implementation ticket
3. `acceptance / regression ticket`

解释如下：

- `test-design ticket`：先定义如何验收
- `implementation ticket`：把验收用例打绿
- `acceptance/regression ticket`：补齐真实浏览器与跨端闭环证据

### 4.3 Ticket 必备信息

除现有 YAML 结构外，本轮 ticket 在内容上必须明确写出：

- 当前 ticket 对应哪个行为单元
- 当前 ticket 属于哪一个流转阶段
- 当前 ticket 影响哪些页面、接口、服务、数据读取链
- 当前 ticket 的上游输入是什么
- 当前 ticket 的下游消费方是谁
- 当前 ticket 明确不负责什么

### 4.4 Ticket 反模式

以下 ticket 写法在本轮视为不合格：

- 标题是一个大功能名，但无法直接转化为测试
- 一个 ticket 同时承接“发起 + 分配 + 审核 + 回看”
- 验收标准只有泛化描述，没有可验证结果
- allowed_paths 很大，但行为边界不清楚
- test_cases 只剩分类标签，没有前置条件和断言

## 5. 验收级 Test Case 标准

### 5.1 目标

本轮 test case 必须达到“别人拿到就能直接执行验收”的程度，不允许只停留在场景标签层。

### 5.2 必填信息

每个 test case 至少必须包含以下信息：

1. `用例目标`
   - 该 case 验证哪个具体行为
   - 对应哪个 AC / 行为单元

2. `前置条件`
   - 当前登录角色
   - 当前账号的正式链资格
   - 当前申请 / 分配 / 课程记录状态
   - 任何必须存在的数据前提

3. `测试数据`
   - 输入参数
   - 关联实体
   - 边界值或非法值

4. `操作步骤`
   - 页面动作或接口动作逐步写清
   - 不允许只写“测试某功能”

5. `预期结果`
   - 页面展示结果
   - 接口返回结果
   - 数据持久化结果
   - 跨页 / 跨端联动结果

6. `禁止结果`
   - 不应该出现的 UI
   - 不应该写入的数据
   - 不应该暴露的权限
   - 不应该提前可见的记录

7. `证据层级`
   - `service`
   - `controller`
   - `frontend contract`
   - `playwright / e2e`
   - `cross-end acceptance`

8. `分支覆盖`
   - 当前 case 覆盖哪个业务分支

9. `边界覆盖`
   - 当前 case 覆盖哪类边界情形

10. `验收判定`
   - 什么条件算通过
   - 什么现象算失败

### 5.3 统一测试设计方法

本轮 test case 统一使用以下设计方法：

- 等价类划分
- 边界值分析
- 决策表
- 状态迁移
- 权限 / 数据边界测试
- 跨端流转断点验证

### 5.4 核心覆盖维度

每个核心行为单元的用例集，至少要覆盖以下类型：

- `happy path`
- `auth_or_data_boundary`
- `business_rule_reject`
- `state_transition`
- `persist_effect`
- `visibility_boundary`

### 5.5 本轮必须覆盖的边界场景

本轮五端课程申请 / 课程记录流转，至少必须覆盖以下边界：

- `null / empty`
- 重复提交
- 越权访问
- 当前状态不允许时的操作
- 审核前不可见
- 审核后可见
- 刷新后仍一致
- 页面空态与受限态
- 主链失败时不得假成功
- carrier route 读取范围不得越过 ownership 边界

## 6. Test Ticket 与 Implementation Ticket 的组织方式

### 6.1 固定组织形态

每个行为单元按以下顺序组织：

1. `test-design ticket`
2. `implementation ticket`
3. `acceptance / regression ticket`

### 6.2 test-design ticket 的职责

`test-design ticket` 负责：

- 定义验收级 test cases
- 明确前置条件、输入、操作、预期结果、禁止结果
- 同步写入 `ticket.test_cases`
- 同步写入模块级 `tasks/testing` 资产
- 产出可供实现直接消费的测试设计

`test-design ticket` 完成时，允许测试仍为红色；但用例定义必须已经细到可执行、可审查、可追溯。

### 6.3 implementation ticket 的职责

`implementation ticket` 只负责把已经定义好的验收用例打绿，不负责重新解释业务语义。

其职责包括：

- 后端主链逻辑实现
- 前端页面职责实现
- 状态映射与边界控制实现
- 对应层级测试从红转绿

### 6.4 acceptance / regression ticket 的职责

`acceptance / regression ticket` 负责：

- 真实浏览器验证
- 跨页 / 跨端联动验证
- 验收证据补齐
- 防止行为单元只在单层测试里通过，但在真实链路中失真

## 7. Done 定义

### 7.1 Ticket Done

一个 ticket 只有在以下条件同时成立时才算 done：

- 对应的验收级 test case 已定义完成
- failing tests 已先出现过
- 当前 ticket 负责的实现已把对应 tests 打绿
- 当前 ticket 的禁止结果被验证未出现
- verification evidence 足以支撑单 ticket 验收

解释：

- `代码写完` 不等于 `ticket done`
- `测试全绿 + 证据完整` 才等于 `ticket done`

### 7.2 Workstream Done

一个 workstream 只有在以下条件同时成立时才算 done：

- 该 workstream 下所有 ticket 已完成
- 页面职责或后端链边界不再重叠
- 关键跨层测试已经打通
- 对应回归 ticket 已通过

### 7.3 Ready for Acceptance

一批 workstream 进入人工验收前，必须同时满足：

- 对应验收级 test cases 全绿
- 至少有一条真实跨端闭环验证通过
- 没有已知 blocker 被留给“验收时再看”
- 页面无技术性报错
- 正式链失败不假成功
- 角色权限边界无歧义

## 8. 与当前五端实施计划的衔接

### 8.1 上层计划

[2026-03-27-five-end-course-application-flow-alignment-implementation-plan.md](/Users/hw/workspace/OSGPrj/docs/plans/five-end-course-flow/2026-03-27-five-end-course-application-flow-alignment-implementation-plan.md) 继续作为本轮需求流转与 workstream 顺序的唯一执行计划。

### 8.2 本文档的作用

本文档作为上层计划的执行门槛，规定：

- workstream 如何拆成可验收行为单元
- ticket 如何组织
- test case 必须细到什么程度
- done 与 ready for acceptance 如何判定

### 8.3 执行顺序

本轮实际执行顺序固定为：

1. 先读取五端 implementation plan，确定当前 workstream
2. 按本文档将该 workstream 拆成：
   - `test-design ticket`
   - `implementation ticket`
   - `acceptance / regression ticket`
3. 按测试先行原则执行
4. 满足 `Ready for Acceptance` 后再进入人工验收

## 9. 本轮试行成功判定

本轮试行若满足以下条件，可视为成功，并可作为后续框架演进输入：

- ticket 粒度能稳定支撑测试先行
- test case 能直接承接人工验收
- implementation ticket 不再需要大量临场补业务判断
- 验收阶段发现的问题显著少于现有粗粒度 ticket 模式
- 至少在 `C1 / C2 / C3` 中有一段被完整跑通

## 10. 结论

本轮不直接修改框架，而是在现有 RPIV 主链上增加一层试行执行标准。

它的核心不是“让 ticket 更小”，而是：

- 让 ticket 变成可验收行为单元
- 让 test case 变成验收资产
- 让 done 的定义从“代码写完”升级为“测试与证据闭环”

若本轮试行有效，后续再将稳定部分沉淀回框架与 guard 体系。
