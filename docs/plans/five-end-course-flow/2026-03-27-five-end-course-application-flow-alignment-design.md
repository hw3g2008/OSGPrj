## 1. 背景

当前围绕学生端、导师端、助教端、班主任端、后台端的“课程申请”和“课程记录”存在多层混用：

- “申请记录”和“课程记录”在部分页面中被当成同一种业务对象理解
- 助教端、班主任端、后台端对“谁能分配导师”的角色边界出现理解漂移
- 学生端页面、跨端原型、现有代码实现没有形成一份可以直接给五端统一对齐的需求口径

这会导致：

- 同一条申请在不同端被当成不同实体理解
- 课程上报、审核、学生回看的链路描述不一致
- 助教、班主任、后台的职责边界产生歧义

本设计的目标，是输出一份可以直接作为五端统一对齐基线的业务定义，先把“业务对象、流转、页面职责、权限边界”收成单一口径，再作为后续实现与原型修正的依据。

- 学生端页面把课程记录详情职责与申请追踪职责混为一页

## 3. 统一业务定义

### 3.1 课程培训申请

学生侧存在两大类课程培训申请：

1. 真实岗位面试辅导申请
2. 模拟类辅导申请

其中“模拟类辅导申请”可以包含：

- 模拟面试
- 人际关系测试
- 期中考试

### 3.2 课程申请记录

课程申请记录是“申请与分配链”的业务对象，表示：

- 学生提出了哪一种辅导需求
- 当前申请处于什么阶段
- 是否已分配导师
- 当前是否仍在辅导中

课程申请记录不是课消记录，也不是导师上课记录。

### 3.3 课程记录

课程记录是“课消链”的业务对象，表示：

- 某一次具体上课的记录
- 导师填写的课程内容与反馈
- 后台审核结果
- 学生评价

一条课程申请记录，可以对应一条或多条课程记录。

## 4. 五端统一流转

### 4.1 申请链

无论是真实岗位辅导，还是模拟类辅导，统一申请链如下：

1. 学生发起课程培训申请
2. 申请进入跨端可见范围
3. 班主任或后台分配导师
4. 导师开始进入上课执行

### 4.2 课程记录链

每一次实际上课，都走统一课程记录链：

1. 导师完成一次授课
2. 导师提交一条课程记录
3. 后台审核课程记录
4. 审核通过后同步到学生端课程记录页
5. 学生查看并评价

### 4.3 多次授课

同一条课程申请记录，可以触发多次授课。

因此：

- 申请记录只有一条主线状态
- 课程记录可以有多条明细

## 5. 五端职责与权限边界

### 5.1 学生端

学生端职责：

- 发起真实岗位面试辅导申请
- 发起模拟类辅导申请
- 查看申请状态、导师信息、摘要进展
- 在课程记录页查看审核通过的课程记录并评价

学生端无权：

- 分配导师
- 审核课程记录

### 5.2 导师端

导师端职责：

- 查看已分配给自己的学员与申请
- 执行上课
- 每次上课后提交课程记录
- 查看课程记录审核状态

导师端无权：

- 分配导师
- 审核课程记录

### 5.3 助教端

助教端职责：

- 查看申请是否进入流程
- 跟进学员进度
- 辅助管理、协同推进

助教端无权：

- 像班主任一样分配导师
- 作为正式分配角色决定导师归属

### 5.4 班主任端

班主任端职责：

- 查看待处理申请
- 分配导师
- 跟踪申请与辅导进度

### 5.5 后台端

后台端职责：

- 查看全局申请
- 分配导师
- 审核课程记录
- 作为全局流程控制端保留最终审核与管理职责

## 6. 三类页面承载

### 6.1 我的求职

学生端 `求职中心 -> 我的求职` 承载：

- 真实岗位面试辅导申请记录
- 真实岗位求职追踪信息
- 与该申请关联的辅导摘要信息

它不是课程记录明细页。

### 6.2 模拟应聘

学生端 `求职中心 -> 模拟应聘` 承载：

- 模拟类辅导申请记录
- 模拟类申请追踪信息
- 与该申请关联的辅导摘要信息

它不是课程记录明细页。

### 6.3 课程记录

学生端 `学习中心 -> 课程记录` 承载：

- 两类申请后产生的统一课程记录
- 每一节课的详细记录
- 导师反馈详情
- 学生评价

它是唯一的课程记录详情页。

## 7. 申请记录与课程记录的展示分工

### 7.1 申请记录页展示内容

申请记录页允许展示：

- 申请内容
- 申请时间
- 当前申请状态
- 导师信息
- 已上课时摘要
- 课程反馈摘要

### 7.2 课程记录页展示内容

课程记录页展示：

- 每次上课的记录
- 上课日期
- 时长
- 课程内容
- 导师反馈详情
- 学生评价

### 7.3 统一规则

因此页面职责是：

- `我的求职` 与 `模拟应聘`：申请追踪 + 结果摘要
- `课程记录`：课消详情 + 学生评价

不允许再把“申请页完全替代课程记录页”或“课程记录页反向承担申请追踪页职责”。

## 8. 五端状态口径

### 8.1 课程申请记录状态

课程申请记录统一使用以下主状态：

- 已提交
- 待分配导师
- 已分配导师
- 辅导中
- 已完成

执行说明：

- 当前一期实现若不引入 schema 变更，`已提交` 与 `待分配导师` 在可执行模型中合并展示为 `待分配导师`
- 该压缩只用于当前实现落地，不改变业务语义层面对“申请已产生”和“等待正式分配”的区分

### 8.2 课程记录状态

课程记录统一使用以下状态：

- 导师已提交
- 后台审核通过
- 学生可见
- 学生已评价

### 8.3 状态约束

- 课程记录在后台审核通过前，不进入学生端课程记录页
- 申请页中的状态与摘要，不替代课程记录审核状态
- 一条申请处于“辅导中”时，可以已经存在多条已审核课程记录
- 根据确认清单，课程记录允许导师、班主任、助教提交；但无论由谁提交，均必须走后台审核后才能进入学生端

## 9. 当前跨端冲突与裁决

### 9.1 助教端分配导师冲突

当前助教端原型中仍出现“分配导师”按钮与弹窗。

但本次已明确裁决：

- 助教端不拥有与班主任同级的导师分配权限
- 正式分配角色只有班主任与后台

因此：

- 助教端现有“分配导师”相关原型元素，应视为待修正偏差
- 后续实现不得以助教端分配按钮为依据保留该权限

### 9.2 申请页与课程记录页边界冲突

当前部分页面与实现中存在：

- 把申请记录直接当课程记录
- 把课程详情职责塞回申请追踪页

本次统一裁决为：

- 申请页可以保留摘要
- 课程记录详情与学生评价必须统一归于 `课程记录` 页

## 11. 验收标准

当且仅当满足以下条件，才算五端对齐完成：

1. 五端都按同一口径区分“课程申请记录”和“课程记录”
2. 助教端不再被理解为正式分配导师角色
3. 班主任端与后台端被明确为正式分配角色
4. 学生端 `我的求职` / `模拟应聘` / `课程记录` 的职责边界清晰
5. 导师端只负责上课与提交课程记录，不承担分配角色

## 12. 结论

本设计收口后的唯一业务理解是：

- 学生有两类课程培训申请：真实岗位辅导、模拟类辅导
- 申请记录走申请与分配链
- 课程记录走授课、提交、审核、回看与评价链
- 班主任与后台拥有正式分配导师权限
- 助教参与流程但不拥有班主任级别的导师分配权
- 学生端申请页展示申请追踪与摘要，课程记录页展示统一课消详情与评价

这份定义稿应作为后续五端继续讨论、实现、测试、修原型时的统一基线。
