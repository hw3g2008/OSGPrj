# OSG 目标业务流转总纲设计稿

日期：2026-03-24  
主题：将 `docs/osg-business-flow-target.md` 收敛为长期可维护的“需求基线总纲”  
状态：已完成设计确认，待进入后续规划阶段

## 1. 背景

当前仓库里已经形成了两类不同性质的事实材料：

- 目标业务真相：基于原型、PRD、业务流程文档整理出来的“业务应该怎样流转”
- 当前实现真相：基于代码、数据库和实际运行行为整理出来的“系统现在实际上怎样流转”

问题不在于资料不足，而在于职责容易混淆。如果 `osg-business-flow-target.md` 同时承载：

- 目标业务定义
- 当前实现差异
- 测试执行顺序
- 临时排障说明

那么它很快会退化成混合笔记，后续既不适合作为产品确认基线，也不适合作为测试和实现的上游文档。

这次设计的目标，是把 `docs/osg-business-flow-target.md` 收敛成一份稳定、清晰、可长期演进的需求基线总纲。

## 2. 设计目标

这份总纲文档需要满足以下目标：

- 作为 5 端首批范围的目标业务真源
- 明确角色边界、业务对象和主链
- 显式暴露待产品确认项
- 为后续 `curl` 串联测试和差异审计提供上游结构
- 与当前实现文档、测试文档形成稳定分层，不互相污染

## 3. 备选方案

### 方案 A：目标、现状、执行三层分离

做法：

- `docs/osg-business-flow-target.md` 只写目标业务
- `docs/osg-data-flow-current-state.md` 只写当前实现
- 后续新增测试文档只写验证步骤和验证结论

优点：

- 需求真源最纯净
- 最适合拿去和产品确认
- 后续差异审计最稳定

缺点：

- 阅读时需要跨文档跳转

结论：

- 推荐采用

### 方案 B：目标文档内直接夹带现状差异

做法：

- 在目标文档每条主链后面直接补“当前实现差异/风险”

优点：

- 一眼就能看到目标和现状的对照

缺点：

- 目标文档会被现状噪音污染
- 容易把实现妥协误看成业务要求

结论：

- 不推荐

### 方案 C：目标正文纯净，文末挂现状附录

做法：

- 正文只写目标业务
- 文末追加“现状差异附录”

优点：

- 比方案 B 更干净

缺点：

- 仍然让一份文档承担双重职责
- 长期维护容易继续膨胀

结论：

- 可作为短期过渡，但不如方案 A 稳定

## 4. 最终决策

正式采用方案 A：

- `docs/osg-business-flow-target.md` 作为目标业务基线总纲
- `docs/osg-data-flow-current-state.md` 作为当前实现真相文档
- 后续再新增执行层文档，承接 `curl` 串联测试与差异验证

总原则：

- 目标层只回答“业务应该怎样”
- 现状层只回答“系统现在怎样”
- 执行层只回答“如何验证现状是否满足目标”

## 5. 目标文档职责边界

### 5.1 这份文档承担什么

`docs/osg-business-flow-target.md` 固定承担 4 个职责：

- 定义 5 端首批范围内的目标业务流转
- 定义角色边界与核心业务对象
- 定义系统主链及其上下游关系
- 显式列出待产品确认项

### 5.2 这份文档不承担什么

`docs/osg-business-flow-target.md` 不承担以下职责：

- 不记录当前代码实现细节
- 不承载接口清单或 `curl` 步骤
- 不记录数据库表、字段、Mapper、控制器行为
- 不替产品裁决未确认业务问题

## 6. 目标文档结构

建议将 `docs/osg-business-flow-target.md` 固化为以下结构：

1. 文档定位  
   说明它是目标业务真源，而不是实现文档或测试脚本

2. 真源与适用范围  
   引用原型、PRD、业务流程文档，并说明本轮首批覆盖范围

3. 业务对象字典  
   统一账号、学员、员工、合同、排期、岗位、求职状态、模拟应聘申请、课程记录、基础数据等术语

4. 角色职责矩阵  
   明确每个角色能发起什么、处理什么、回看什么

5. 目标主链总览  
   先给出系统围绕哪些主链运转

6. 分链详解  
   每条主链按统一模板书写

7. 待产品确认项  
   用统一字段管理歧义和冲突

8. 配套文档  
   只挂链接，不混入正文

额外调整建议：

- 将“当前测试口径”改名为“首批适用范围”或“本轮覆盖范围”
- 将“测试意义”统一改名为“验证重点”或“业务价值”
- 将“由业务反推的测试顺序”从正文主结构降级为附录或配套文档入口

## 7. 主链统一模板

每条主链必须使用统一模板，避免风格漂移。

推荐模板如下：

1. `主链名称`
2. `业务目的`
3. `起点角色与入口`
4. `前置条件`
5. `核心流转步骤`
6. `关键状态`
7. `下游消费角色`
8. `结束条件`
9. `本链边界`
10. `待确认依赖`

使用原则：

- 只写业务动作与状态，不写实现接口和数据库
- 入口必须明确到“角色 + 页面”
- 结束条件必须可判断
- 边界必须明确这条链不负责什么

对现有 A-F 主链的建议调整：

- 为每条主链补齐 `前置条件`
- 为每条主链补齐 `结束条件`
- 为每条主链补齐 `本链边界`
- 将“正常业务流转”统一改名为 `核心流转步骤`
- 将“测试意义”统一改名为 `验证重点`

## 8. 待确认项管理机制

第 7 节“待产品确认项”不应只是普通列表，而应升级为受控的歧义管理区。

每个待确认项统一包含以下字段：

- `问题`
- `冲突来源`
- `影响范围`
- `建议默认策略`
- `是否阻塞`
- `确认结果`

使用规则：

- `问题` 必须可裁决
- `冲突来源` 必须指出冲突的真源文档或原型页面
- `影响范围` 必须明确影响哪几条主链、哪几个端
- `建议默认策略` 仅用于临时分析，不代替产品拍板
- `是否阻塞` 仅区分：
  - `阻塞主链测试`
  - `不阻塞，可后置确认`
- `确认结果` 在产品裁决后回填，并同步更新正文

总原则：

- 未确认的业务问题，不允许被当前实现默认吞掉
- 未确认但不阻塞的项，可按建议默认策略继续分析
- 未确认且阻塞主链的项，不进入正式串联测试

## 9. 文档分层与联动方式

建议建立稳定的三层文档关系：

### 9.1 目标层

文件：

- `docs/osg-business-flow-target.md`

职责：

- 只回答“业务应该怎样流转”

### 9.2 现状层

文件：

- `docs/osg-data-flow-current-state.md`

职责：

- 只回答“当前实现实际上怎样流转”

### 9.3 执行层

文件：

- 后续新增的测试文档、差异审计文档、`curl` 清单

职责：

- 只回答“如何验证现状是否满足目标”

### 9.4 单向引用规则

- 目标层只引用原型、PRD、业务流程真源
- 现状层可以引用代码、表、运行行为，并回指目标层主链名
- 执行层同时引用目标层和现状层
- 不允许因为实现方便而直接修改目标层定义

### 9.5 维护规则

- 产品确认了业务变化：先改目标层
- 代码实现变化但业务不变：只改现状层
- 测试覆盖策略变化：只改执行层

## 10. 主链稳定 ID 方案

为减少后续沟通成本，建议给每条主链加稳定 ID：

- `FLOW-A` 账号与主数据准备链
- `FLOW-B` 岗位发布与曝光链
- `FLOW-C` 求职状态流转链
- `FLOW-D` 模拟应聘链
- `FLOW-E` 课程记录链
- `FLOW-F` 个人中心与排期链

这些 ID 后续可被以下内容复用：

- 现状文档
- `curl` 测试清单
- 差异审计报告
- 缺陷单和阻塞项

## 11. 建议的后续动作

在不改实现的前提下，下一步建议按以下顺序推进：

1. 按本设计重构 `docs/osg-business-flow-target.md`
2. 将待确认项升级为受控确认区
3. 为主链补充稳定 ID 和统一模板
4. 在目标层文档中仅保留对现状层文档的短链接
5. 基于目标层和现状层，单独编写执行层测试文档

## 12. 本次设计结论

这次设计最终确定：

- `docs/osg-business-flow-target.md` 要成为“需求基线总纲”
- 它必须保持目标业务纯度，不混入实现噪音
- 它需要稳定结构、统一主链模板和显式歧义管理机制
- 它与 `docs/osg-data-flow-current-state.md` 的关系应为并列分层，不互相吞并
