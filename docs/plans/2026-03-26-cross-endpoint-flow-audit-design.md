# OSG 跨端数据流第二轮审计设计稿

日期：2026-03-26  
主题：以“可写且跨端流转”的业务链路为核心，执行第二轮跨端数据流与审批必要性审计  
状态：设计已确认，待进入审计执行

## 1. 背景

首轮审计已经在 [docs/osg-cross-endpoint-interaction-audit.md](/Users/hw/workspace/OSGPrj/docs/osg-cross-endpoint-interaction-audit.md) 中识别出一批跨端断链、错端点和字段映射问题，但这些发现仍然偏“问题清单”，还不够像一份稳定可复用的链路审计方法。

当前仓库的主要风险不是单个页面实现偏差，而是以下三类系统性问题可能在多个模块中重复出现：

- 写入与审批读取不在同一张表或同一份真相数据上
- 前端 API、后端 Controller、Service、Mapper、数据库字段之间存在错路或脏映射
- 需求上本应审批的链路没有审批闭环，或本不需要审批的链路被错误实现成审批流

因此，本轮不再以“按页面扫问题”为主，而改为以“按数据流链路审问题”为主。

## 2. 目标

本轮目标固定为三件事：

1. 找出所有“可写且跨端流转”的高风险链路中，是否仍存在与首轮同类的断链、错路、脏映射和能力缺口。
2. 对每条链路额外回答一个产品问题：这条链路按需求到底应不应该审批。
3. 产出一份可复查、可追踪、可用于后续修复验收的链路矩阵，而不仅是散点式问题列表。

## 3. 范围与边界

### 3.1 本轮纳入范围

本轮只覆盖“会写数据、会跨端流转、会被别的角色继续处理”的链路：

1. `Profile / Change Request / 审批流`
2. `课程与课时提交 -> Admin 审核 -> 结果回写/展示`
3. `岗位与求职记录` 的创建、分配、审核、展示
4. `登录 / 忘记密码 / 资料接口` 这类身份与恢复链路
5. `地区 / 方向 / 状态` 等字典驱动字段在跨端之间的 key/value 一致性

### 3.2 本轮排除范围

以下内容不作为本轮主目标：

- 纯静态展示页
- 只有本端读写、不跨角色流转的数据
- 纯样式、文案、命名风格一致性问题
- 没有写入口、没有审批入口、没有跨端读取的只读页面

## 4. 方案选择

### 4.1 方案 A：只复查 Profile / Change Request / 审批流

优点：

- 聚焦最快
- 最容易继续抓到与首轮同类的问题

缺点：

- 容易漏掉课程、岗位、身份恢复等其他跨端写链路中的同类结构性问题

### 4.2 方案 B：全量扫五端所有创建-审批-展示-回写链路

优点：

- 覆盖最全

缺点：

- 成本高
- 容易把真实断链问题和低优先级整洁性问题混在一起
- 首轮产出不够聚焦

### 4.3 方案 C：先扫所有“可变更且会跨端流转的数据链路”，暂不扫纯展示页

优点：

- 比只看 Profile 更广，能覆盖同类系统性问题
- 比全量五端更收敛，能把注意力集中在真正会出事故的链路上

缺点：

- 需要先定义统一的链路审计模板和严重度标准

### 4.4 最终选择

采用方案 C。

原则如下：

- 先深后广
- 先查真实会写数据的链路，再查只读展示
- 先判断业务流是否成立，再讨论命名和风格问题

## 5. 审计方法

本轮所有链路统一按同一套“数据流模板”审计，而不是按页面印象或单点代码判断。

每条链路至少检查以下 7 个点：

1. 写入口：哪个端、哪个页面、哪个前端 API 发起写操作
2. 请求契约：前端字段名、payload 结构、关键枚举值
3. 后端入口：命中的 Controller / Service / Mapper
4. 持久化目标：真正写到哪张表、哪几个字段
5. 跨端读取：下一个角色从哪里读，读的是不是同一份数据
6. 审批闭环：是否有 `list / approve / reject / review` 全链路
7. 最终回写：审批或处理完成后，正式数据是否真的回写到最终展示源

## 6. 问题分类与严重度

### 6.1 问题分类

本轮发现的问题统一分成 4 类：

- `断链`：有写入，但后续无人可读、不可审、不可回写
- `错路`：请求打到了错误端点、错误角色边界、错误数据源
- `脏映射`：前后端 key、字段、枚举、表结构不一致
- `能力缺口`：流程存在，但缺字段、缺入口、缺状态处理

### 6.2 严重度

- `P0`：提交后核心流程无法继续，或审批/回写必崩
- `P1`：流程能走，但数据源错、角色边界错、结果不可信
- `P2`：流程基本可走，但一致性差，需要人工兜底
- `P3`：主要是命名、风格、体验问题

## 7. 审批必要性判定

本轮会把“技术上有没有审批接口”和“按需求应不应该审批”拆开判断，不把二者混为一谈。

每条链路都要额外给出一个审批判定：

- `直接写入`：需求允许本端直接改正式数据，不需要申请、不需要人工审核
- `发起申请 + 自动生效`：需要保留申请或痕迹，但不需要人工 approve/reject
- `发起申请 + 必须审批`：必须进入待审核队列，且应存在明确的 `approve/reject`
- `不是审批，而是流转/分配`：属于状态推进或角色交接，不能误判成审批流

同时，对每条链路还要给出两类偏差结论：

- `该审批但没审批`
- `其实不该审批，却被实现成审批或待审核`

## 9. 审计分组

本轮按以下 5 组链路执行：

### 9.1 个人资料与变更申请

覆盖：

- Student
- Mentor
- Lead-Mentor
- Assistant
- Admin 对应审核或查看入口

重点：

- 写入数据源是否与审核数据源一致
- 哪些字段应直接生效，哪些字段应进入审核
- 审核通过后是否回写到正式展示源

### 9.2 课程与课时链路

覆盖：

- Mentor / Lead-Mentor / Assistant 提交
- Admin 审核
- Student 回看与评价

重点：

- 提交记录是否真正进入后台审核链
- 审核结果是否回写到学生端和统计口径
- 审核是否为需求强制链

### 9.3 岗位与求职流转

覆盖：

- Student 创建或申请
- Admin 审核
- Lead-Mentor / Assistant 分配或查看
- Mentor 只读或消费

重点：

- 原始申请事实与工作流投影是否混淆
- 通过、驳回、分配是否是同一条链路的连续状态
- 审批与分配是否被错误混用

### 9.4 登录、忘记密码与身份恢复

覆盖：

- Student
- Mentor
- Lead-Mentor
- Assistant
- Admin 相关恢复能力

重点：

- 端点是否打到正确角色命名空间
- 恢复流程是否错误复用其他角色入口
- 这类链路通常不属于审批，但属于高风险错路检查对象

### 9.5 字典与枚举驱动字段

覆盖：

- 地区
- 主攻方向 / 子方向
- 状态值
- 其他前后端共享枚举

重点：

- 前端 value 和后端存储值是否一致
- 展示值、提交值、数据库值是否混淆
- 是否因 key/value 不一致导致跨端无法继续流转

## 11. 统一结论格式

每条链路最终输出统一包含：

- `链路名`
- `写入口`
- `持久化目标`
- `下游读取方`
- `是否应审批`
- `是否已具备审批闭环`
- `最终状态/正式数据回写点`
- `判定结果`
- `问题类型`
- `严重度`

## 12. 通过标准

### 12.1 通过

满足以下条件时，该链路视为通过：

- 需求定义清楚
- 写读同源
- 角色边界正确
- 需要审批的链路具备完整闭环
- 审批或处理结果能回写到正式展示源

### 12.2 存疑

符合以下任一条件时，链路记为存疑：

- 需求文档没有明确审批属性
- 代码中存在双真值或临时投影，暂时无法确定主真相
- 需求与实现都不够明确，不能直接下结论

### 12.3 失败

符合以下任一条件时，链路记为失败：

- 出现断链
- 出现错路
- 出现脏映射
- 出现错误审批策略
