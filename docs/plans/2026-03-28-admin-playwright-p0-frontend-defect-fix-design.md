# Admin Playwright P0 前端缺陷修复设计

日期：2026-03-28  
主题：admin 端 P0 Playwright 验收缺陷高标准前端修复设计

## 1. 背景与修订原因

本次任务只处理 admin 前端真实问题，不修改 `admin-test/` 下的真源 TSV 和 MD。  
修复范围限定在 `osg-frontend/packages/admin/src`。

设计在首轮收敛后，用户进一步提高了边界要求：

- `gap` 类问题交给另一个 agent 处理，本设计不再混入 `gap`
- 当前页面里已经存在的可见按钮、链接、菜单项，只要交互存在坏态，就应视为前端实现问题
- 不允许把页面内的真实坏态归入“真源缺口”后跳过
- 仍然不允许为了过验收硬改业务代码去掩盖账号、夹具、共享环境问题

因此，本设计是对首版方案的高标准修订版，目标是把“真实 defect”与“gap / 环境问题”彻底拆开，再按批次做最小风险修复。

## 2. 设计目标

本次设计目标不是一次性重构整个 admin，而是在最小风险下完成 P0 前端缺陷闭环：

1. 只修真实前端 defect，不掩盖账号、夹具、环境问题。
2. 只要页面里已有可见入口，且交互行为坏掉，就纳入前端修复范围。
3. `gap` 独立排除，由其他 agent 处理，不与当前批次混修。
4. 所有修复都有对应测试和定向回归。
5. 不引入新的命名双轨、伪交互或共享组件回归风险。

## 3. 问题分类标准

### 3.1 真实前端 defect

满足以下任一项即进入本次修复范围：

- 页面可见按钮、链接、菜单项未绑定
- 入口绑定错误，触发了错误弹窗或错误动作
- `data-surface-trigger` / `data-surface-id` 缺失、错绑、命名错误
- modal 可打开但无法按预期关闭，或关闭 / 取消链路异常
- 入口文案、位置、行为与页面语义或公开契约不一致
- 页面存在 `message.info(...)`、空函数、coming soon、后续版本接入等假动作
- 页面有可见操作，但提交、成功提示、失败提示与真实结果不一致
- 页面已有入口，但后续交互链路残缺

这里的核心约束是：

- 先看页面当前是否真的暴露了入口
- 只要入口可见且交互坏掉，就按真实前端 defect 处理

### 3.2 明确排除项

以下问题不进入本次前端代码修复：

- `gap`
- 缺少 `admin_limited_without_permission` 账号
- 缺少安全回滚夹具或稳定测试夹具
- 共享环境不适合真实提交
- 页面本身没有该控件，只是验收资产侧存在定位问题

### 3.3 真源 / 验收侧缺口

只有在以下情况下，才继续归入真源 / 验收侧缺口，而不是前端 defect：

- 页面本身根本没有该控件
- 页面也不存在对应坏态
- 问题只来自验收资产无法稳定定位

如果页面中已经能看到该控件，则不再以“真源缺口”为由排除。

## 4. 方案比较

### 4.1 方案 A：真实 defect 批次化推进

把 `gap` 与非前端问题全部剥离，只处理已发现且能在当前前端源码中落到真实坏态的 defect。  
按模块批次推进，每批固定走：分诊 -> 测试补强 -> 最小修复 -> 定向回归 -> 结果记录。

优点：

- 边界清楚
- 风险最低
- 最适合你后续回归

缺点：

- 节奏更严格
- 不追求一次性全站标准化

### 4.2 方案 B：共享契约统一后再逐页扫

先把所有 modal / trigger / close / cancel / disable 规则统一，再回到各页面补行为。

优点：

- 一致性更高
- 共享问题能一次解决

缺点：

- 共享改动面更大
- 更容易引入新回归

### 4.3 方案 C：按 defect 清单顺序逐条修

完全贴着验收清单顺序推进，一条条关闭。

优点：

- 对照最直观

缺点：

- 上下文切换频繁
- 共享根因容易被拆散

### 4.4 结论

采用方案 A 作为主流程：

- `gap` 全部排除出当前批次
- 真实 defect 按模块分批修
- 共享组件只在确认是根因时才动
- 不做全 admin 大重构

## 5. 总体推进框架

### 5.1 设计关口

通过 `brainstorming` 先固化：

- 范围
- 分类标准
- 批次
- 质量 gate
- 回归门槛

未经批准，不进入新的实现修改。

### 5.2 缺陷分诊

基于以下输入建立修复清单：

- `admin-test/2026-03-28-admin-playwright-run-results.tsv`
- `admin-test/2026-03-28-admin-playwright-defects.md`
- 受影响 admin 前端源码
- 现有 `vitest` 规范

输出：

- 真实前端 defect 清单
- 非前端问题清单
- `gap` 排除清单

### 5.3 实施闭环

每个修复批次固定执行：

1. 把 defect 映射到具体页面 / 组件 / 入口
2. 先补或更新 `vitest`
3. 再做最小代码修复
4. 跑批次级定向回归
5. 做完成前验证

### 5.4 输出闭环

最终必须给出：

- 已修 ManifestItem / defect
- 改动文件
- 新增或更新测试
- 剩余非前端问题
- 未验证项与残余风险

## 6. Skill / 能力调度设计

`brainstorming` 只负责设计关口。  
实施时按环节调用不同能力。

### 6.1 `brainstorming`

用途：

- 锁定范围、边界、批次与 gate

Gate：

- 未批准不进入新的实现修改

### 6.2 缺陷分诊

用途：

- 判断哪些是当前页面里的真实前端 defect
- 明确剥离 `gap`

Gate：

- 只有真实前端 defect 才进入实施

### 6.3 `test-driven-development`

用途：

- 先把缺陷失败形态固化为 `vitest`

Gate：

- 没有测试边界，不做代码修复

### 6.4 实施修复

用途：

- 以最小改动修复真实前端问题

原则：

- `surface contract` 优先
- 共享组件是根因才动
- 页面局部问题只做局部补丁
- 不为账号 / 夹具 / 环境 / `gap` 修改业务逻辑

### 6.5 定向回归

默认工具：

- `vitest`
- `webapp-testing`，仅用于补真实交互验证

### 6.6 `verification-before-completion`

用途：

- 在宣布完成前做最终核验

Gate：

- 没有测试证据，不宣称修复完成

## 7. 修订后的批次设计

### 7.1 批次 1：权限管理与后台用户剩余真实 defect

这是当前实现的下一个主批次，优先级最高。

目标：

- 收完角色管理、后台用户管理、权限相关页面里还没闭环的真实 defect
- 重点复核并修复 close / cancel / trigger / modal 契约问题

建议纳入：

- `ADM-PW-ROLE-007`
- `ADM-PW-ADMINUSER-034`
- 以及同页中页面已暴露入口、但交互存在坏态的真实 defect

处理原则：

- 只要页面已有可见入口，且链路残缺，就不再按 `gap` 跳过
- 如果源码真实没坏，不为了配合验收做无意义改动
- 若共享 modal / surface 行为是根因，则做最小共享收口

### 7.2 批次 2：career 范围内的真实 defect

范围：

- `views/career/positions/**`
- `views/career/student-positions/**`
- `views/career/job-overview/**`
- `views/career/mock-practice/**`

目标：

- 扫描这些页面里已经存在的可见入口
- 只处理真实前端坏态，不处理 `gap`

优先类型：

- 可见未绑定
- 错绑
- 假动作
- 空动作
- 假成功提示

### 7.3 批次 3：teaching 范围内的剩余真实 defect

范围：

- 当前范围内仍未闭环的 teaching 侧真实 defect
- 包括 `views/teaching/class-records/index.vue` 之外仍存在的页面级坏态

目标：

- 继续按“页面已有入口但行为坏掉”标准清理
- 不保留假动作和假成功

## 8. 公共质量约束

### 8.1 入口层

- 所有相关入口必须挂在真实可点击语义元素上
- 不把 `data-surface-trigger` 挂在无语义 wrapper
- 行内样本入口按需补 `data-surface-sample-key`

### 8.2 弹窗层

- 每个公开 modal 都必须有唯一 `data-surface-id`
- 右上关闭、取消、确认行为一致
- 不允许孤儿 modal、重复 modal ID、不可达 modal

### 8.3 行为层

- 取消只负责关闭，不产生副作用
- 确认才触发提交
- 成功 / 失败提示必须与真实结果一致
- 不允许“看起来成功但实际上没做事”

### 8.4 兼容层约束

- 不新增长期双命名契约
- 不用兼容逻辑掩盖错绑
- 不顺手做无关重构

## 9. 测试与回归设计

### 9.1 第一层：缺陷对应测试

每个修复点必须新增或更新 `vitest`。

批次 1 重点：

- `src/__tests__/roles.spec.ts`
- `src/__tests__/users.spec.ts`
- 如动到共享 surface，再跑 `src/__tests__/overlay-surface.spec.ts`
- 如动到 layout/menu，再跑 `src/__tests__/layout.spec.ts`

批次 2 重点：

- 对实际 touched 页面补对应 spec
- 如改动 modal/surface 共享行为，再补 `overlay-surface.spec.ts`

批次 3 重点：

- 对实际 touched teaching 页面补对应 spec
- `class-records.spec.ts` 继续承担假动作回归

覆盖重点：

- `data-surface-trigger`
- `data-surface-id`
- 关闭 / 取消 / 入口绑定
- 文案与可点击性
- 假动作替换为真实行为或明确禁用态

### 9.2 第二层：批次级定向回归

每批至少满足：

- 当前批次新增 / 更新 spec 全部通过
- 同批次相关旧 spec 不回归失败
- 如触及共享 overlay 契约，对应共享 spec 必跑
- `pnpm --filter @osg/admin build` 通过

### 9.3 第三层：必要时浏览器级抽查

只在结构测试不足以证明真实交互时，才使用 `webapp-testing` 做抽查。

典型场景：

- 菜单入口打开弹窗
- modal 右上关闭与取消链路
- 假动作改禁用态后的真实页面表现

## 10. 完成判定

宣布某一批 defect 完成前，必须同时满足：

1. 每个已修 defect 都能映射到具体代码改动和测试。
2. 当前批次定向回归通过。
3. 剩余问题都已明确归类为账号 / 夹具 / 环境或 `gap`。
4. 汇报中显式区分“已修”和“未修但不在当前前端修复范围”。

## 11. 风险控制

本次明确避免：

- 把页面里的真实坏态错误归入 `gap`
- 全局大重构
- 长期双命名兼容层
- 用前端假提示掩盖环境问题
- 在没有测试兜底的前提下改共享 modal 契约
- 修改 `admin-test/` 真源文件
