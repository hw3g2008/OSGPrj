# 学生端正式数据流收口设计稿

日期：2026-03-27  
状态：设计已确认，待进入实施计划  
范围：学生端首批保留页面的数据来源、写入链路、错误处理与跨端一致性收口

## 1. 背景

当前学生端首批保留页面已经完成了导航白名单收口，但数据层仍处于“主链 + 影子链 + 兼容吞错”并存状态：

- 页面表面可用，但部分写操作在正式主链失败时会静默回退或假成功
- 部分页面读取仍混合主表、学生独立表和前端演示态
- 同一业务概念在学生端、班主任端、后台端未完全共享同一条正式链路

这与本轮落地目标不一致。本轮目标不是“页面先跑起来”，而是把当前对用户可见的学生端页面收口成真正可交付状态：

1. 现有可见页面不能出现技术性报错
2. 对的就是对的，错的就是错的，不允许假成功
3. 可见数据必须能解释清楚正式来源
4. 学生端写入与班主任/后台读取必须符合正式业务流转

## 2. 业务真相

本轮设计确认以下业务口径：

- 学生存在两类辅导申请场景
  - 真实岗位辅导申请
  - 模拟应聘辅导申请
- 两类申请后续都遵循同一授课与记录闭环
  - 学生发起申请
  - 班主任/助教分配导师
  - 导师授课，可多次
  - 导师提交多次课程记录
  - 后台审核
  - 学生统一回看
- 学生端课程记录页是唯一课程记录回看入口
- 求职状态仍是全局唯一主状态
- 基础数据仍是统一配置源
- 排课与课程分配仍属于首批正式业务前置链路

## 3. 设计目标

本轮固定目标如下：

1. 学生端 5 个保留页面全部只展示正式可解释的数据
2. 去掉正式链失败后的静默降级和假成功
3. 把“模糊课程申请”收口成两个明确业务意图
4. 让学生端与班主任/后台针对同一业务对象共享同一正式主链
5. 统一课程记录回看口径

## 4. 非目标

本轮明确不做以下事情：

- 不恢复已隐藏的非首批页面
- 不对五端所有页面做一次性全量数据重构
- 不新建一个全新的“统一课程申请主表”
- 不重做整套权限体系
- 不为了兼容历史影子链继续保留假成功

## 5. 推荐方案

采用“方案 B：双申请主链收口 + 统一课程记录回看”。

核心思路：

- 学生端把辅导申请拆成两个明确业务意图
  - 真实岗位辅导申请：归入岗位/求职主链
  - 模拟应聘辅导申请：归入模拟应聘主链
- 两类申请都不再通过语义模糊的混合入口写影子表
- 导师授课后形成的正式课程记录统一由学生端课程记录页回看
- 正式身份未打通时，页面进入受控失败或受限态，而不是偷偷降级

## 6. 页面与数据流设计

### 6.1 岗位信息

页面允许展示两类内容：

- 公共岗位池
- 学生自添待审核岗位

但两类内容必须显式区分：

- 公共岗位相关操作
  - 投递
  - 进度更新
  - 真实岗位辅导申请
  必须统一写入正式求职主链
- 学生自添岗位在审核通过前仅显示为待审核条目
- 审核通过后才进入公共岗位正式曝光链

页面不再允许以下行为：

- 公共岗位写主链失败后静默成功
- 自添岗位在待审核阶段伪装成正式公共岗位
- 前端用硬编码默认业务值填充关键字段

### 6.2 我的求职

页面只展示正式求职主状态。

具体要求：

- 学生在岗位信息页做的投递、进度更新、真实岗位辅导申请，必须在这里回看到同一份正式状态
- 班主任端和后台端读取的求职主状态，必须与学生端读取同源
- 不再允许主链写失败后，学生页从影子表读出“看起来正常”的结果

### 6.3 模拟应聘

页面只承担模拟应聘业务：

- 模拟应聘辅导申请
- 模拟应聘状态展示
- 模拟应聘相关历史回看

页面不再保留“既像课程申请又像模拟应聘申请”的模糊入口。

本轮将现有混合入口改成两个明确意图之一：

- 真实岗位辅导申请：放在岗位/求职侧发起
- 模拟应聘辅导申请：放在模拟应聘页发起

### 6.4 我的课程

页面是唯一课程记录回看入口。

统一规则：

- 无论前面来自真实岗位辅导申请还是模拟应聘辅导申请
- 只要导师授课并形成课程记录
- 只要后台审核通过
- 学生都统一在课程记录页回看

页面中每条课程记录应带来源语义标签，例如：

- 真实岗位辅导
- 模拟应聘辅导

这样既统一回看入口，也保留来源可解释性。

### 6.5 基本信息

资料页继续保留统一视图，但写入必须明确分流：

- 即时生效字段
- 需审核后生效字段

设计要求：

- 即时字段只在正式可允许的字段范围内直接更新
- 审核字段统一走正式变更申请链
- 若当前账号无法进入正式资料链，不允许降级到旧 pending 表后仍向前端返回成功

## 7. 正式链前置条件

所有会影响正式业务流转的学生端页面，在写正式主链前必须满足：

- 当前登录账号可解析到正式 `student_id`
- 当前账号满足进入正式业务链的必要条件

若不满足，系统进入受控状态：

- 返回明确业务失败
- 前端展示可理解的受限提示
- 不抛技术栈报错
- 不走 shadow fallback 伪装成功

可接受的提示类型包括：

- 当前账号未进入正式业务链
- 当前岗位仍在审核中，暂不可发起正式辅导
- 当前暂无已审核课程记录
- 当前申请已提交，等待班主任或助教分配导师

## 8. 错误处理原则

本轮“不报错”的定义不是“什么都别提示”，而是：

- 不出现技术性炸错
- 不出现假成功
- 不出现静默降级

禁止行为：

- 主链失败但前端提示成功
- 主链失败后改读影子表继续展示成功结果
- 直接暴露技术性异常信息
- 用前端硬编码业务值伪装真实数据

允许行为：

- 明确业务失败
- 明确业务空态
- 明确等待中状态
- 明确受限态

## 9. 跨端一致性要求

本轮验收不接受“学生端讲一套、班主任/后台讲另一套”。

必须满足：

- 学生端看到的求职主状态，与班主任/后台看到的是同一条正式状态链
- 学生端看到的模拟应聘申请状态，与班主任/后台读到的是同一条模拟应聘主链
- 学生端课程记录页展示的审核通过记录，与后台审核结果保持一致

## 10. 测试与验收

### 10.1 验收结果

本轮通过标准固定为：

1. 学生端保留页无技术性报错
2. 所有正式写入动作无假成功
3. 可见数据都能说明正式来源
4. 两类辅导申请链都能闭环
5. 课程记录统一回看成立
6. 学生端与班主任/后台口径一致

### 10.2 必测闭环

至少验证以下两条完整链路：

1. 真实岗位辅导申请
   - 学生对真实岗位发起辅导申请
   - 班主任/助教分配导师
   - 导师多次授课
   - 导师多次提交课程记录
   - 后台审核
   - 学生课程记录页统一回看

2. 模拟应聘辅导申请
   - 学生发起模拟应聘辅导申请
   - 班主任/助教分配导师
   - 导师多次授课
   - 导师多次提交课程记录
   - 后台审核
   - 学生课程记录页统一回看

### 10.3 测试分层

- 后端服务测试
  - 正式身份存在时必须写入主链
  - 正式身份缺失时必须明确失败
  - 不允许主链失败后静默成功
- 前端页面测试
  - 正常态
  - 空态
  - 受限态
  - 写失败提示
  - 不出现硬编码假业务值
- 跨端链路验证
  - 学生端发起
  - 班主任/后台读取
  - 课程记录审核
  - 学生统一回看

## 11. 实施顺序建议

建议实施顺序如下：

1. 先收口正式链前置条件与错误处理规则
2. 再收口岗位/求职主链的写入与读取一致性
3. 再拆分模拟应聘页中的模糊申请入口
4. 再统一课程记录回看来源与标签
5. 最后收口资料页正式写链与失败表现

## 12. 设计结论

本轮不再接受“为了页面可用先混着跑”的策略。

正式收口口径为：

- 真实岗位辅导申请走岗位/求职主链
- 模拟应聘辅导申请走模拟应聘主链
- 两类申请后的授课记录统一进入课程记录审核与学生回看链
- 页面失败要真实失败，成功要真实成功
- 可见页面的数据必须来自可解释的正式流程
