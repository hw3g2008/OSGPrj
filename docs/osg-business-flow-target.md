## 2. 当前测试口径

本轮测试只围绕用户已明确要求的首批范围：

- 学生端：登录、忘记密码、岗位信息、我的求职、模拟应聘、课程记录、基本信息
- 导师端：登录、忘记密码、课程记录、学员求职总览、模拟应聘管理、基本信息、课程排期
- 班主任端：登录、忘记密码、岗位信息、学员求职总览、模拟应聘管理、学员列表、课程记录、基本信息、课程排期
- 助教端：登录、忘记密码、岗位信息、学员求职总览、模拟应聘管理、学员列表、课程记录、基本信息、课程排期
- 后台管理：登录、忘记密码、权限配置、后台用户管理、基础数据管理、学员列表、合同管理、导师列表、导师排期管理、岗位信息、学生自添岗位、学员求职总览、模拟应聘管理、课程记录、操作日志、个人设置

## 3. 核心业务对象

后续所有 `curl` 测试，都应围绕这些业务对象建链，而不是按接口名散点冒烟：

- 账号：登录账号、密码找回、角色权限
- 学员：基础资料、求职方向、账号状态、归属班主任/导师/助教
- 员工：导师、班主任、助教、后台用户
- 合同：合同金额、课时、有效期、续签
- 排期：导师/班主任/助教可用时间，后台可监控与协调
- 岗位：岗位库、展示状态、招聘周期、公司/地区/部门
- 求职状态：未申请、已申请、面试中、已获 Offer、已拒绝
- 模拟应聘申请：模拟面试、人际关系测试、期中考试
- 课程记录：授课申报、审核、评价、结算
- 基础数据：学校、方向、地区、行业、招聘周期、角色菜单等配置项

## 4. 角色职责边界

### 4.1 学生端

- 查看岗位库
- 收藏岗位、手动补充岗位、申请辅导、记录求职进度
- 查看“我的求职”中的面试安排与辅导进度
- 发起模拟应聘申请
- 查看课程记录并评价导师/班主任/助教提交的课程
- 维护自己的基本信息

### 4.2 导师端

- 接收已分配的辅导任务和模拟应聘任务
- 查看自己负责学员的求职进度
- 上报课程记录与反馈
- 维护自己的基本信息与课程排期

### 4.3 班主任端

- 查看岗位库
- 查看并跟进自己管理学员的求职进度
- 处理模拟应聘申请并分配导师
- 查看学员列表
- 查看自己的课程申报以及自己管理学员的课程记录
- 维护自己的基本信息与课程排期

### 4.4 助教端

- 查看岗位库
- 查看并协助跟进自己管理学员的求职进度
- 处理模拟应聘申请并分配导师
- 查看学员列表
- 查看自己的课程申报以及自己管理学员的课程记录
- 维护自己的基本信息与课程排期

### 4.5 后台管理端

- 配置后台角色权限与后台用户
- 创建学员、导师、班主任、助教等主数据
- 建立和维护合同
- 监控并必要时调整导师排期
- 维护岗位库，处理学生自添岗位
- 全局查看求职总览和模拟应聘记录
- 审核课程记录
- 记录操作日志，管理个人设置

## 5. 目标主链

### 5.1 主链 A：账号与主数据准备链

起点：后台管理端  
终点：5 端都能以真实角色账号进入各自业务页面

正常业务流转：

1. 后台在“权限配置/后台用户管理”中准备后台角色和后台账号。
2. 后台在“导师列表”中录入导师、班主任、助教，并开通对应角色账号。
3. 后台在“学员列表”中新增学员，填写基础资料、求职方向、导师配置。
4. 后台在“合同管理”中为学员创建首份合同，后续可续签。
5. 导师、班主任、助教在“课程排期”填报自己的可用时间；后台在“导师排期管理”中监控、催填、代填或调整。
6. 学生、导师、班主任、助教、后台用户分别从各自门户登录；若忘记密码，则从各自门户发起找回。

测试意义：

- 这是所有业务链的前置主数据。
- 如果这条链不成立，后面的岗位、模拟应聘、课程记录都只是无主数据的空跑。

### 5.2 主链 B：岗位发布与曝光链

起点：后台管理端“岗位信息”  
终点：学生端、班主任端、助教端能看到正确岗位库

正常业务流转：

1. 岗位管理员或超级管理员收集招聘信息。
2. 后台新增岗位，或通过模板批量上传岗位。
3. 系统按“公司/岗位/招聘周期”进行排重。
4. 后台设置展示开始/结束时间、状态和分类。
5. 岗位上线后，学生、班主任、助教可在各自“岗位信息”页浏览。
6. 岗位结束后，若无学员关联则自动删除；若有学员关联则转为归档或过期，后台可重新激活。

测试意义：

- 这是求职链和模拟应聘链的共同上游。
- 学生侧“手动添加岗位”如果属于正式能力，还会形成一条“学生补充岗位 -> 后台审核/入库”的支链。

### 5.3 主链 C：求职状态流转链

起点：学生端“岗位信息/我的求职”  
终点：班主任、助教、导师、后台在各自视角看到同一份学员求职进度

正常业务流转：

1. 学生浏览岗位，并可收藏、标记已投递、记录阶段。
2. 学生在岗位页或“我的求职”页更新自己的求职状态。
3. 学生在申请辅导或申请某类课程时，同时填写公司、岗位、当前阶段和面试时间。
4. 班主任、助教在自己的“岗位信息”或“学员求职总览”中查看所管理学员的求职进度，并可代学员更新状态。
5. 后台在“学员求职总览”中查看全局进度，并在需要时修改任意学员状态。
6. 导师在自己的“学员求职总览”中查看已分配给自己的辅导对象的进度，用于后续辅导。

测试意义：

- 这是跨端共享最多的一条链。
- 如果业务要求成立，那么学生、班主任、助教、后台、导师看到的应当是同一条学员求职主线，只是可见范围不同。

### 5.4 主链 D：模拟应聘链

起点：学生端“模拟应聘”  
终点：导师反馈回到学生、班主任、助教、后台

正常业务流转：

1. 学生在“模拟应聘”页发起申请。
2. 申请类型包括：
   - 模拟面试
   - 人际关系测试
   - 期中考试
3. 学生填写申请内容、目标场景，必要时填写希望导师人数或面试背景。
4. 班主任或助教在“模拟应聘管理”中查看待处理申请，并分配导师与预约时间。
5. 后台管理端可以在全局“模拟应聘管理”中查看全部申请，也可以处理待分配申请。
6. 导师在导师端“模拟应聘管理”中看到分配给自己的任务，完成辅导并提交反馈。
7. 学生在“模拟应聘”记录里看到已安排、进行中、已完成状态，并能查看导师反馈。
8. 班主任、助教、后台在各自页面查看反馈和结果，用于继续推进求职或课程安排。

测试意义：

- 这是最标准的“学生发起 -> 管理侧分配 -> 导师执行 -> 学生/管理侧回收结果”的闭环链。

### 5.5 主链 E：课程记录链

起点：导师/班主任/助教提交课程记录  
终点：学生评价 + 后台审核 + 合同/结算侧产生后续结果

正常业务流转：

1. 课程先基于求职辅导、模拟应聘或其他课程申请被安排。
2. 授课完成后，导师、班主任或助教在各自“课程记录”页提交课程申报。
3. 课程申报包含：
   - 学员
   - 辅导类型
   - 课程内容
   - 上课日期与时长
   - 课程反馈
4. 后台在“课程记录”页审核课程申报。
5. 审核通过后，课程记录进入后续课时统计或结算流程。
6. 学生在学生端“课程记录”中看到新记录，并对课程进行评价。
7. 课程记录结果反哺到：
   - 学生课程历史
   - 导师端历史申报
   - 班主任/助教管理视角
   - 后台审核与统计视角
   - 合同剩余课时或结算视角

测试意义：

- 这是“教学中心”最核心的跨端主链。
- 也是合同课时与求职辅导结果之间的连接点。

### 5.6 主链 F：个人中心与排期链

起点：导师/班主任/助教/学生在个人中心维护资料  
终点：业务分配、过滤和展示使用最新资料/排期

正常业务流转：

1. 学生在“基本信息”中维护个人资料、学业信息、求职方向。
2. 导师、班主任、助教在“基本信息”中维护自己的资料。
3. 导师、班主任、助教在“课程排期”中维护可用时间。
4. 后台在“导师排期管理”中查看是否填报、是否可协调、是否需要催填或代填。
5. 排期数据被班主任/助教/后台在分配课程或模拟应聘时使用。
6. 学生的求职方向又会影响岗位曝光、筛选和推荐。

测试意义：

- 这是很多“看起来是展示字段”的页面，其实会影响岗位筛选、导师匹配、排期分配的上游数据链。

## 6. 由业务反推的测试顺序

如果后续要做严谨 `curl` 串联，建议按以下顺序，而不是按页面顺序逐个点：

1. 账号与权限准备
2. 学员/员工/合同/排期主数据准备
3. 岗位发布
4. 学生求职状态发起与更新
5. 班主任/助教查看并跟进
6. 模拟应聘申请与导师分配
7. 导师执行辅导并提交课程记录
8. 学生查看课程记录并评价
9. 后台审核课程记录
10. 回查后台、班主任、助教、导师、学生 5 端视角是否一致

## 7. 已确认业务结论

以下结论已按产品回复整理；未单独回复项默认同意建议结论，后续测试统一按此口径执行。

### 7.1 导师参与求职中心，但权限受限

- 导师参与“已分配学员的求职总览”和“模拟应聘管理”。
- 导师不参与岗位库维护。
- 导师不修改学员求职状态，默认只查看、辅导和反馈。

### 7.2 学生自添岗位属于正式支链

- 学生可在学生端手动添加岗位。
- 学生自添岗位需经后台审核。
- 审核通过后进入公共岗位库，其他学生可见。

### 7.3 求职状态为全局唯一主状态

- 学生、班主任、助教、后台操作的是同一份学员求职主状态。
- 学生在岗位页、我的求职页、课程申请等入口填写的状态，应统一回写到这份主状态。
- 导师默认只读，不参与主状态修改。

### 7.4 排课管理属于课程记录链的前置主链

- 本轮测试默认包含“学生申请课程 -> 班主任/助教分配导师 -> 导师授课 -> 提交课程记录”完整链路。
- 不再把课程记录视为可独立脱离排课主链的孤立模块。

### 7.5 模拟应聘默认覆盖三种类型

- 模拟应聘统一包含：
  - 模拟面试
  - 人际关系测试
  - 期中考试
- 本轮测试默认三类都纳入覆盖范围。

### 7.6 课程记录允许三类角色提交

- 导师、班主任、助教都可以提交课程记录。
- 后台负责审核。
- 审核通过后同步到学生端和相关管理视角。

### 7.7 学员允许配置多个班主任、多个助教

- 学员与班主任、助教按多对多关系理解。
- “我的学员”及相关查询口径，默认按最终分配关系取数。

### 7.8 基础数据管理为全平台统一配置源

- 岗位、学校、主攻方向、子方向、地区、招聘周期、课程类型等基础项，统一由后台基础数据管理维护。
- 各端下拉项、筛选项、标签项默认同步使用同一套基础数据。

### 7.9 合同是进入正式业务链的前置条件

- 学员进入正式求职、模拟应聘、课程记录主链前，默认需先具备首份合同。
- 合同是正式建档、课时和后续统计结算的前置基础。

### 7.10 黑名单纳入本轮测试范围

- 黑名单学员和黑名单导师可正常登录。
- 黑名单用户默认不能进入求职中心。
- 限制范围默认不扩展到课程记录等其他模块，除非产品后续追加规则。

## 8. 后续动作

基于上述已确认结论，后续 `curl` 测试按下面顺序推进：

- 先按本文件第 5 节主链拆测试清单
- 正式执行稿见 `docs/osg-curl-chain-checklist.md`
- 为每条主链补“起点接口、状态变化、下游回查”脚本
- 按统一业务口径回查后台、班主任、助教、导师、学生 5 端视角是否一致
