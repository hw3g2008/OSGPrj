# Superpowers 强制触发规则

本规则**不替代** RPIV 主流程。当任务走 RPIV 时（/brainstorm、/split、/next 等），按 RPIV 规范执行。
当任务是**小需求 / bug 修复 / 小重构**（走 `/fix`、`/feat-small`、`/refactor` 或无 slash command 的直接请求）时，以下 3 条强制触发规则**必须**遵守。

---

## R1. Bug 修复前必须走 Systematic Debugging

**触发条件**（任一命中即触发）：
- 用户报告："xxx 不工作" / "xxx 报错" / "xxx 异常" / "xxx 出问题了"
- 用户提供报错栈、截图、日志
- 用户说"测试失败" / "页面白屏" / "接口 500"
- 任何以"修一下"、"查一下"、"为什么"开头的问题

**强制动作**：
1. **禁止**直接修改代码
2. **必须**先调用 `systematic-debugging` skill，走完 4 阶段（Reproduce → Investigate → Root Cause → Fix Plan）
3. 输出根因 + 修复方案，等用户确认后才能改代码（与"代码修改前必须确认"规则叠加生效）

**违规判定**：
- 收到 bug 描述后，未调用 `systematic-debugging` 就调用 `edit`/`multi_edit` = 违规
- 提出修复方案前未明确说明"根因是 xxx" = 违规

**例外**：用户明确说"不用查了，直接改 X 为 Y"——这种情况按用户指令执行，但仍需"代码修改前确认"。

---

## R2. 声称"完成"前必须 Verification

**触发条件**：
- AI 准备回复"修好了 / 完成了 / OK 了 / 已修复 / done"
- AI 准备说"应该可以了 / 试试看"
- 任何代码/配置修改后的总结性回复

**强制动作**：
1. **必须**先调用 `verification-before-completion` skill
2. **必须**提供至少一条可复现的验证证据，三选一：
   - 跑测试：实际执行 `pnpm test` / `mvn test` / `pytest` 等，贴退出码和关键输出
   - 跑命令：`curl` / `grep` / 构建命令，贴实际输出
   - 提供用户可自己验证的命令（当无法在环境中运行时），明确标注"请你执行以下命令验证"
3. **禁止**仅凭代码 diff 就声称完成

**违规判定**：
- 说"修好了"但没有任何验证输出、没有命令、没有测试结果 = 违规
- 仅贴修改后的代码片段作为"证据" = 违规（代码能写不代表能跑）

**例外**：用户明确说"不用验证，我自己测"——按用户指令。

---

## R3. 连续失败 2 次必须 When Stuck

**触发条件**：
- 同一 bug / 同一问题，AI 已经尝试修复 2 次但未成功
- 用户回复"还是不行" / "还有问题" / "报了新错" 2 次及以上
- AI 自己判断"我陷入循环了"

**强制动作**：
1. **禁止**第 3 次继续按同思路盲改
2. **必须**调用 `when-stuck` skill 重新分流
3. 输出三件事：
   - **原假设是什么**（前 2 次我以为根因是 X）
   - **为什么被证伪**（证据表明不是 X）
   - **新方向是什么**（根据 when-stuck 建议切换到的新思路）
4. 可能的新方向包括但不限于：
   - 调用 `root-cause-tracing` 反向追踪调用栈
   - 调用 `inversion-exercise` 翻转假设
   - 调用 `dispatching-parallel-agents` 多角度并行排查
   - 请用户提供更多上下文（日志、截图、复现步骤）

**违规判定**：
- 第 3 次回复仍在重复前 2 次的思路（只是小参数调整）= 违规
- 没说明"前两次假设是什么、为什么错"就继续改 = 违规

---

## 自检清单（AI 每次回复前必须过一遍）

- [ ] 这是 bug 报告吗？是 → 我调用 systematic-debugging 了吗？
- [ ] 我在说"完成/修好了"吗？是 → 我有实际验证输出吗？
- [ ] 这是对同一问题的第 3+ 次尝试吗？是 → 我调用 when-stuck 重新分流了吗？

---

## 与现有规则的关系

| 已有规则 | 本规则如何叠加 |
|---|---|
| 代码修改前必须确认 | R1 在"确认"之前再加一道"根因分析" |
| 不考虑效率只考虑质量 | R1/R2/R3 都是为质量服务，强制不走捷径 |
| 先定位根本原因再修复 | R1 把这条规则具体化为"必须走 systematic-debugging 4 阶段" |
| 只做最小修改避免过度工程 | R1 输出的"修复方案"必须是最小改动 |

本规则是**原有规则的加强版**，不冲突。
