# /validate-doc — 文档先行校验流程

**适用场景**：
- 方案文档 / 需求文档 / 变更说明 / 设计文档
- 文档先行的工作习惯：先写清楚想做什么，校验通过后才实施
- 中等及以上规模的变更（小 bug 不需要，直接 `/fix`）

**原则**：
- **只校验文档，不动代码**
- 出阻塞级问题必须停下，用户决策后再推进
- 4 轮思辨全跑，不省略（质量优先）
- 🚨 **严格限定在文档声明的范围内校验**，禁止提出新需求、未来扩展、架构演进等超范围问题

---

## 🚨 Scope Guard — 范围约束（最高优先级）

**本工作流只回答一个问题**：文档所声明的目标/方案是否**完整、一致、可实施**？

### 必须在范围内的问题（✅ 合法）
- 文档内部假设之间是否矛盾
- 方案是否覆盖了**文档声称要解决的**所有问题
- 代码引用（行号/文件名/字段名）是否真实
- 术语/字段名/数字前后是否一致
- 方案里有没有**多余**或**遗漏**的改动
- 决策点之间是否冲突
- 验收标准是否可度量、是否覆盖所有声称的修复

### 禁止提出的问题（❌ 超范围）
- **新需求**："其实还应该做 X"（即便 X 合理，也不在本次范围）
- **未来扩展**："如果将来支持 EUR 呢" / "如果运行 5 年呢" / "1000 倍规模会怎样"
- **架构演进**："这个字段其实可以删" / "这个设计扩展性差"
- **展示层优化**："用户看到的体验应该更好"（除非文档明确涉及）
- **代码重构**："这个函数可以抽取" / "可以更优雅"
- **跨文档问题**：其他模块/其他文档的问题

### 判断口诀
提出每个问题前，**先问自己一遍**：
> "如果这个问题不处理，文档声称的**本次**目标还能达成吗？"
- **否** → 在范围内，合法
- **是**（只是未来可能有问题） → 超范围，丢弃

**Round 2 Scale 的特别约束**：
- 不问"1000 倍 / 5 年 / 扩展币种"这类问题
- Scale 只用来检查"方案在文档声明的量级内是否成立"（通常就是当前量级，所以 Round 2 对修复类文档**大概率空转**，这是正常的）

**Round 1 Inversion 的特别约束**：
- 只翻转文档**显式列出**的假设
- 不翻转"做不做这件事本身" / "这个架构对不对"

**输入**：一个文档路径（可以是正在编辑的文件）

---

## Step 1 — 读懂文档，确认理解

1. `read_file` 完整读完文档
2. 输出理解摘要：
   ```
   ## 我的理解
   - 目标: [一句话]
   - 核心变更: [3-5 条]
   - 关键决策点: [列出每个"选 A 不选 B"的地方]
   - 隐含假设: [作者可能没写但依赖的前提]
   - 明确标记"本期不做": [列出]
   ```
3. ⛔ **停下等用户确认**："理解对吗？有没有漏掉的意图？"

用户确认后才进入 Step 2。

---

## Step 2 — 4 轮思辨校验（强制全跑）

每轮读对应 SKILL.md 后执行：`~/.codeium/windsurf/skills/<skill-name>/SKILL.md`

### Round 1: Inversion Exercise — 翻转核心假设

调用 `inversion-exercise` skill。对文档里的**每个核心假设**问："如果相反会怎样？"

示例问题：
- 假设"用户填写时自行按汇率换算" → 翻转："如果用户不会换算 / 换算错了呢？"
- 假设"程序不做汇率换算" → 翻转："什么情况下必须做？"
- 假设"无存量数据" → 翻转："上线后出现存量怎么办？"

输出：
```
## Round 1: Inversion 发现
- 假设 A → 翻转问题 → [是否发现漏洞]
- ...
```

### Round 2: Scale Game — 范围内规模压力

调用 `scale-game` skill，但**严格限定在文档声明的规模范围内**。

**允许问的问题**：
- 文档声明的当前规模下（如"本期 10 个币种内"），方案是否成立
- 文档声明的边界条件（如"0 金额" / "单数字月日" / "老数据"），方案是否覆盖

**禁止问的问题**：
- 未来扩展到 N 倍会怎样（超范围）
- 运行 N 年后会怎样（超范围）
- 如果规模扩展到文档未声明的维度（超范围）

**如果本轮对修复类文档无有效发现，直接写"无发现（修复范围内的规模已被覆盖）"，不要硬造问题。**

输出：
```
## Round 2: Scale 发现（范围内）
- 场景 → 方案是否 break → [发现]
- 或：无发现
```

### Round 3: Simplification Cascades — 方案内部是否最小化

调用 `simplification-cascades` skill。**限定在本次修复方案内部**找冗余。

**允许问的问题**：
- 修复方案里有没有**多余的改动**（改 A 和改 B 实际是一回事）
- 有没有**重复的逻辑**（同一处判断写了两次）
- 有没有**可合并的分支**（本次修复范围内的分支）

**禁止问的问题**：
- 现有字段/架构/函数能不能砍（属于架构重构，超范围）
- "这个设计本来就有问题"（超范围，本次是修 BUG 不是重构）

输出：
```
## Round 3: 简化机会
- 可砍掉: [项] — 理由: [...]
- 可合并: [项] — 理由: [...]
- 不能砍: [项] — 即使表面冗余也必要，理由: [...]
```

### Round 4: Preserving Productive Tensions — 决策间是否冲突

调用 `preserving-productive-tensions` skill。**限定在文档已列出的决策点之间**找冲突。

**允许问的问题**：
- 决策 A 和决策 B 是否互相矛盾（例：假设 X 与 D11 逻辑冲突）
- "本期不做" 的事项是否与"本期要做"的事项**逻辑上不能共存**
- 某个决策是否**实际上需要先做另一个决策**

**禁止问的问题**：
- "这个决策其实还有第三个选项"（超范围，不做需求延伸）
- "被压下的反对意见" 若不是文档内的决策冲突，属超范围

输出：
```
## Round 4: 被忽略的权衡
- 决策点 → 被压下的反对意见 → 是否值得保留选项
- ...
```

---

## Step 3 — 一致性校验

**内部一致性**：
- 术语统一（例：同一概念有没有多个名字）
- 字段名一致（PRD 写 A，后端 SQL 写 B？）
- 数字/路径/文件名是否对得上
- 前后矛盾（Section 3 说做，Section 8 说不做？）

**外部一致性**：
- 引用的代码位置是否存在（`grep` 验证）
- 引用的其他文档是否存在
- 与现有代码结构是否冲突

输出：
```
## 一致性发现
- 内部: [问题列表]
- 外部: [问题列表]
```

---

## Step 4 — 输出校验报告（分级）

汇总 Step 2~3 所有发现，按影响分级：

```
# 校验报告: <文档路径>

## 🔴 阻塞级（必须处理才能实施）
1. [问题] — 来源: Round X / 一致性 — 建议改动: [...]
2. ...

## 🟡 建议级（最好处理）
1. [问题] — 理由 — 建议
2. ...

## 🟢 信息级（仅提示风险）
1. [风险] — 后续需关注
2. ...

## 结论
- 阻塞级数量: N
- 是否可以进入实施: [是/否]
```

---

## Step 5 — ⛔ 用户决策门

**不要自动改文档，不要自动进实施。** 输出报告后停下，等用户选：

- **阻塞级全改**：用户回复"按建议改"或逐条指示 → 调用 `/implement-fix-plan` 或手动更新文档 → 改完**重新跑 `/validate-doc`** 直到无阻塞级
- **部分接受**：用户明确说"X 不改，标记为本期不做" → 在文档追加"本期不做"说明 → 再跑一次校验
- **全盘重写**：用户决定推翻重来 → 结束本次 `/validate-doc`

**禁止**：
- 擅自修改文档
- 擅自进入代码实施
- 忽略阻塞级问题推进到下一步

---

## 失败升级

如果 Step 2 的 4 轮走下来发现**根本假设错了**（例：文档想解决 A 问题，实际根本问题是 B），不要硬凑修复建议。

直接告诉用户："文档的定位可能错了，建议用 `brainstorming-ideas-into-designs` 重新拉一次 Socratic 讨论，而不是在现有文档上打补丁。"

这是 `simplification-cascades` 的推论：有时最大的简化是**不做这个**。
